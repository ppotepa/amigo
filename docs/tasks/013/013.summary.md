# TASK 013: Value Curves and 2D Particle Emitters

## Goal

Add two generic engine features:

- `Curve1d` value curves in `amigo_math` for reusable nonlinear `0..1 -> value` mapping.
- `ParticleEmitter2D` as a generic 2D FX system driven by value curves.

Asteroids is only the first consumer. Engine code must stay game-agnostic.

## Scope Correction

Implement in phases, not as one large patch:

1. Add `Curve1d` and use it in `FreeflightMotion2D`.
2. Add scene YAML parsing/hydration for motion response curves.
3. Add `amigo_2d_particles` with continuous circle emitters.
4. Add scene command/runtime system/render extraction/WGPU rendering.
5. Add Rhai `world.particles`.
6. Replace Asteroids static thrust triangle with a `ParticleEmitter2D`.

Defer `burst`, full `Quad`/`Line` rendering, and asteroid explosions unless the continuous thruster path is complete.

## Key Decisions

- Curves live in `amigo_math`; no runtime plugin.
- `Curve1d::sample()` clamps input to `0..1`.
- Empty custom curves return `1.0`, because most runtime uses are multipliers.
- Particles are not `SceneEntity` instances. Emitters are scene components; runtime particles are lightweight draw data.
- Existing particles may finish their lifetime after emitter deactivation, but inactive emitters do not spawn new particles.
- Motion response curves affect analog input magnitude. Keyboard `0/1` input needs a separate future input ramp feature for temporal easing.

## Done When

- Old scenes without curves still parse and behave as before.
- `FreeflightMotion2D` supports optional response curves from YAML.
- `ParticleEmitter2D` can emit fading/growing circle particles.
- Asteroids thruster uses particles controlled by Rhai intensity.
- Tests cover curves, motion curves, scene hydration, particles, render extraction, and Rhai particle control.

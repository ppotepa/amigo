# TASK 003 - Playground Sidescroller 2D as a vertical slice

## Status

Planned

## JIRA

`AMIGO-002`

## Type

Feature / Engine Integration / Playground Demo

## Priority

High

## Goal

Add a new mod:

`playground-sidescroller`

This mod should be the first real 2D gameplay-oriented vertical slice for Amigo. It is not a full game. It is a feature-integration playground that proves the engine can compose multiple domains into one working loop:

```text
scene YAML
-> scene hydration
-> runtime plugins and domain services
-> Rhai scene logic
-> 2D physics and platformer movement
-> tilemap rendering and collision
-> sprite and animation state
-> screen-space UI HUD
-> generated audio
-> WGPU rendering
```

The hosted command that must work at the end:

```bash
cargo run -p amigo-app -- --hosted --mod=playground-sidescroller --scene=vertical-slice --dev
```

## Why This Task Exists

Current playgrounds validate isolated capabilities:

1. sprite rendering
2. text rendering
3. spritesheet animation
4. Rhai scene scripting
5. screen-space UI
6. basic 3D

That is good for foundation work, but not enough to prove that the architecture can compose multiple systems into a coherent gameplay loop.

This task is meant to be the first practical proof that Amigo can support a small, playable, inspectable 2D scene built from reusable engine domains rather than scene-specific hacks.

The sidescroller is intentionally chosen because it forces real integration of:

1. content assets
2. tilemap data
3. collisions
4. movement
5. jump logic
6. camera follow
7. animation state
8. HUD state
9. event flow
10. audio feedback

It is small enough to remain an engine task, but broad enough to reveal architectural problems if the domain boundaries are wrong.

## Core Architectural Rules

1. Plugin-first architecture only.
2. No gameplay hacks in `app` host.
3. No renderer-specific gameplay logic.
4. No platformer logic inside `Physics2D`.
5. No audio logic inside platformer logic.
6. `Rhai` orchestrates behavior and reacts to events, but does not replace engine subsystems.
7. Scene schema extensions must not break existing scenes.
8. Every new domain should have a narrow service and its own runtime plugin.

## Asset Sourcing, Licensing, and Provenance Supplement

This supplement is part of Task 003 and is mandatory.

The main rule:

For `playground-sidescroller`, external visual assets are allowed only when their license is clear, safe, and documented inside the repo. Prefer `CC0`. Audio is not sourced from external SFX packs in this task; audio is generated by the engine through the generated-audio domain.

### Why This Supplement Exists

This vertical slice is also the first real test of Amigo's external visual content pipeline:

1. image decode
2. metadata-driven interpretation
3. texture upload
4. sprite and tileset usage
5. hot reload
6. reproducible provenance

We want fast progress without drawing all art ourselves, but we do not want licensing chaos in the repo.

### Preferred Asset Sources

#### 1. Kenney - preferred source

Kenney is the first choice for this task.

Reasons:

1. consistent style
2. good prototyping quality
3. pack-oriented distribution
4. often includes tiles, props, collectibles, and character material
5. simple licensing model, typically `CC0`
6. easy provenance documentation

Recommended use:

1. one main Kenney pack for the core tileset
2. optional second Kenney pack only if needed for supplements like collectibles or background
3. avoid mixing too many styles

#### 2. itch.io - allowed after license verification

itch.io is allowed only if the asset page clearly states a safe license.

Requirements:

1. explicit license
2. license allows repo/demo use and redistribution where needed
3. author and source page can be recorded in repo
4. asset is not merely marked `free` without rights information

Preferred itch.io search constraints:

1. game assets
2. platformer
3. free
4. `CC0` or equivalent clear permissive license

#### 3. OpenGameArt - cautious fallback

OpenGameArt is allowed, but only carefully.

For this task:

1. prefer `CC0` only
2. avoid mixed-license packs
3. avoid unclear provenance
4. avoid assets with disputed or ambiguous usage history

If a non-`CC0` OpenGameArt asset is ever considered, that requires an explicit decision and documented attribution rules. That is outside the default path for Task 003.

#### 4. Internal placeholders - acceptable supplement

If a tiny asset is missing, a minimal local placeholder is acceptable.

Examples:

1. finish flag
2. debug collider marker
3. fallback coin
4. simple player capsule/square

Such files must be documented as locally created project assets.

### Accepted and Rejected Licenses

Accepted without extra discussion:

1. `CC0`
2. local project-created placeholder assets
3. procedurally generated audio assets

Accepted only after explicit verification:

1. `CC-BY`
2. `OGA-BY`
3. other permissive licenses with clear repo/demo redistribution rules

Rejected for this task:

1. assets with no license
2. assets marked only as `free`
3. assets with unclear redistribution rights
4. assets forbidding modification if we must crop, split, or repack them
5. assets from unclear marketplace/account-bound licenses
6. game rips
7. fan art based on third-party IP
8. assets with unknown authorship
9. external SFX packs for jump, coin, hurt, or finish

Key rule:

`free` is not enough. Every external visual asset must have a real license trail.

### One Visual Source Preferred

For visual consistency:

1. prefer one main visual pack
2. add a second pack only if necessary
3. keep style, resolution, and pixel density coherent

Do not randomly mix unrelated visual styles.

### No External Audio Rule

Do not import external `jump.wav`, `coin.wav`, `hurt.wav`, or similar files in this task.

All audio feedback for the vertical slice must be represented as generated-audio metadata:

1. `jump.yml`
2. `coin.yml`
3. `hurt.yml`
4. `level-complete.yml`
5. `proximity-beep.yml`

This is part of the core engine goal of the slice.

### Asset Key Rules

Scene and metadata files must use asset keys, never absolute paths.

Correct examples:

1. `playground-sidescroller/textures/player`
2. `playground-sidescroller/tilesets/platformer`
3. `playground-sidescroller/audio/coin`

Incorrect examples:

1. `C:/Users/.../Downloads/player.png`
2. `../../Downloads/tileset.png`
3. `assets/random/coin.wav`

### Metadata Rules

Every real visual asset that needs interpretation should have a metadata file.

Typical required metadata cases:

1. player sprite sheet
2. coin sprite or sprite sheet
3. finish marker if not a trivial static sprite
4. tileset
5. generated audio clips

Expected examples include:

1. `player.yml` with frame size, grid, and animation ranges
2. `platformer.yml` with tile size, grid, tile ids, and collision meaning
3. `coin.yml` with optional spin animation
4. generated-audio YAML for one-shots and realtime sources

### Asset Selection Process

Use this process:

1. identify one or two candidate packs
2. verify license on the source page
3. prefer `CC0`
4. download the archive
5. keep the original archive/file name in provenance notes
6. copy only the needed files into the mod
7. rename imported files to `kebab-case`
8. create metadata YAML
9. document provenance
10. verify scene uses mod-relative asset keys
11. run bootstrap
12. confirm assets are registered, loaded, prepared, and not failing

### Allowed Modifications

Allowed:

1. rename
2. crop
3. split
4. repack into spritesheet
5. scale with nearest-neighbor if needed for pixel art
6. remove unused transparent space
7. add metadata
8. derive debug overlays when documented

If a committed file is modified relative to the downloaded original, provenance must state that clearly.

### Do Not Do This

1. do not commit entire downloaded archives
2. do not commit whole asset packs if only a few files are used
3. do not commit assets without provenance notes
4. do not mix multiple licenses silently
5. do not use assets found through image search without a real source page
6. do not use sprite rips from commercial games
7. do not use external audio files for the generated-audio use cases

### Fallback Strategy

If the real image loader is not ready at an early implementation step:

1. metadata and asset layout should still be prepared
2. temporary debug placeholders are acceptable during intermediate development
3. final Task 003 acceptance still requires real visual assets in the hosted demo

If the chosen pack lacks an animated player:

1. select a second clearly licensed visual source
2. or use a documented local placeholder temporarily

If the chosen pack lacks a coin or finish marker:

1. add a small documented local placeholder
2. or add a second clearly licensed `CC0` supplement

## Scene Script Responsibilities

`scene.rhai` for the vertical slice should do orchestration, not low-level solver logic.

### `on_enter()`

1. initialize score
2. initialize HUD
3. start realtime audio source if needed

### `update(dt)`

1. read input
2. drive platformer controller
3. update HUD debug info
4. update realtime audio params
5. reset player after fall if needed

### `on_event(topic, payload)`

1. coin collected -> score++, HUD update, play coin sound
2. jump -> play jump sound
3. player died / hurt -> play hurt sound, maybe reset
4. finish reached -> HUD message, finish sound, stop realtime source

## Frame Flow

The target runtime flow per frame:

1. `InputState` captures keys
2. scene script converts input into movement intent
3. `PlatformerController2D` consumes intent
4. `PlatformerController2D` updates desired velocity
5. `Physics2D` resolves movement against static colliders
6. `Physics2D` updates transform and grounded/collision state
7. `PlatformerController2D` updates facing and animation state
8. `Sprite2D` chooses current frame or animation
9. triggers emit events
10. `Rhai` reacts with HUD/audio/game state updates
11. audio systems process commands and source params
12. WGPU renders tilemap, sprites, text, and UI

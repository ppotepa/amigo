# TASK 001 - WGPU playground 2D/3D: scene-centric `HELLO WORLD`

## Status

Planned

## Goal

Implement the first two target playgrounds for Amigo:

1. `playground-2d`
   - one YAML scene
   - rotating square
   - `HELLO WORLD` text
   - `Left/Right` rotates the square
2. `playground-3d`
   - one YAML scene
   - rotating cube
   - `HELLO WORLD` text in 3D
   - `Left/Right` rotates the cube around Y
   - `Up/Down` rotates the cube around X

The playgrounds should validate the real engine path:

```text
YAML scene -> scene hydration -> engine services -> Rhai API -> WGPU rendering -> input update loop
```

## Context

The project already has part of the required stack:

1. `playground-2d` already has `sprite-lab.yml` and `text-lab.yml`
2. `playground-3d` already has `mesh-lab.yml` and `material-lab.yml`
3. the engine already hydrates YAML scenes into `SceneCommand`
4. domains already exist for `Sprite2D`, `Text2D`, `Mesh3D`, `Material3D`
5. runtime already registers plugins for assets, hot reload, scene, window, input, WGPU, 2D sprite, 2D text, 3D mesh, 3D material, modding, and Rhai
6. `amigo-render-wgpu` already exists as the active backend, even if part of it is still foundational
7. Rhai already exposes basic scene/asset queue APIs

Use the existing mechanisms if they fit. Do not duplicate them. Extend the current flow instead of bypassing it with playground-specific hardcode.

## Architecture Rules

### 1. Mod is scene-centric

The mod describes scenes, assets, and scripts. The mod should not know WGPU internals.

Allowed in a mod:

```yaml
scene:
  id: main
entities:
  - id: square
    name: playground-2d-square
    transform2:
      rotation_radians: 0.0
    components:
      - type: Sprite2D
      - type: ScriptController
```

Not allowed in a mod:

```rust
// hardcoded WGPU draw pipeline inside playground mod
device.create_render_pipeline(...)
```

### 2. Engine exposes reusable APIs

If rotation is needed, add generic engine mechanisms:

1. `Transform2D`
2. `Transform3D`
3. transform mutation by entity name/id
4. input state
5. render commands

Do not add APIs like:

```rust
rotate_playground_square_only()
```

Prefer:

```rust
scene.transforms.rotate_2d(entity, delta);
scene.transforms.rotate_3d(entity, Vec3::new(dx, dy, dz));
```

### 3. Rhai is a facade over engine contracts

Rhai should not touch service internals directly. It should call stable APIs such as:

```rhai
rotate_entity_2d("playground-2d-square", angle);
rotate_entity_3d("playground-3d-cube", pitch, yaw, roll);
is_key_down("ArrowLeft");
```

### 4. WGPU is only the renderer backend

WGPU receives prepared engine data:

1. sprite draw commands
2. text draw commands
3. mesh draw commands
4. material/light/camera state
5. transform snapshots

WGPU does not decide what a mod or scene is.

### C. Transform runtime API

Extend `crates/engine/scene` with transform mutation after hydration.

Required API:

```rust
impl SceneService {
    pub fn transform_of(&self, entity_name: &str) -> Option<Transform3>;
    pub fn set_transform(&self, entity_name: &str, transform: Transform3) -> bool;
    pub fn rotate_entity_2d(&self, entity_name: &str, delta_radians: f32) -> bool;
    pub fn rotate_entity_3d(&self, entity_name: &str, delta_euler: Vec3) -> bool;
}
```

Minimal intent:

```rust
pub fn rotate_entity_2d(&self, entity_name: &str, delta_radians: f32) -> bool
pub fn rotate_entity_3d(&self, entity_name: &str, delta: Vec3) -> bool
```

#### C2. Draw command synchronization

Current domain services keep cloned transforms in draw commands. That means scene mutation can drift from what the renderer sees.

Target rule:

1. draw commands carry `entity_id`
2. renderer prefers current transform from `SceneService`
3. domain commands keep render data, but scene owns authoritative transform state

Minimal transition:

```rust
pub struct SpriteDrawCommand {
    pub entity_id: SceneEntityId,
    pub entity_name: String,
    pub sprite: Sprite,
    pub transform: Transform2,
}
```

Renderer should prefer current scene transform and only fall back to stored transform.

### G. Playground scripts

#### G1. `mods/playground-2d/scripts/main.rhai`

## Most Important Technical Decision

Entity transform must become mutable runtime state after scene hydration.

Without that, square/cube content may load correctly from YAML but cannot be rotated cleanly through input and Rhai without breaking architecture.

Because of that, the real core of Task 001 is not "add two YAML files", but complete the full loop:

```text
scene YAML -> hydration -> runtime transform state -> Rhai/input -> render snapshot -> WGPU frame
```

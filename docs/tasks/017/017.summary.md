# TASK 017: playground-2d-terrain-duel Foundations + Mod Plan

## Cel

Dodać mod `playground-2d-terrain-duel` jako testowy poligon dla dużego, niszczalnego świata 2D, split screena, projectile-heavy broni i per-player UI.

Task ma zbudować reużywalne fundamenty engine tam, gdzie naprawdę są potrzebne, ale nie rozdmuchiwać architektury o crate’y dla każdej warstwy gameplayu.

## Najważniejsza korekta architektoniczna

Nie wszystko potrzebuje osobnego crate’a w MVP.

Wymagają osobnych domen engine:

- `destructible_surface_2d`
- `viewport_2d`

`destructible_surface_2d` nie jest “terrain systemem”. To generyczny feature dla runtime-edytowalnych powierzchni/masek 2D. Terrain duel używa go jako terenu, ale ten sam mechanizm ma pasować też do ścian, asteroid, lodu, osłon, barier, warstw scorch i edytorowych masek.

Mogą zostać `mod-first` albo jako cienkie rozszerzenia istniejących systemów:

- pickupy;
- environment presets;
- weapon config i match flow;
- część projectile orchestration, jeśli obecny runtime `motion_2d` + `physics_2d` + `particles_2d` wystarcza.

## Pickupy: zmiana względem pierwotnego planu

Pickupy mają być zwykłymi encjami sceny z komponentami używanymi już przez obecny silnik:

- transform;
- sprite/vector;
- collider/trigger;
- tagi lub lekki marker danych;
- skryptowa reakcja na zebranie.

Nie dodawać osobnego crate `pickup_2d` tylko po to, żeby opakować prosty flow `entity enters trigger -> script applies effect -> entity hides/disables -> optional respawn`.

Jeśli później pickup spawn/respawn/query zacznie się powtarzać między modami, wtedy można wydzielić generyczny crate. Na tym etapie to byłoby przedwczesne.

## Zakres

- nowy mod `mods/playground-2d-terrain-duel`;
- nowy crate `crates/2d/destructible-surface`;
- nowy crate `crates/2d/viewport`;
- sceny: `menu`, `match`, `result`, `terrain-lab`, `split-screen-lab`, `weapon-lab`;
- YAML-first config dla destructible surfaces używanych jako terrain, weapons, materials, generatorów i environment presetów;
- Rhai jako cienka warstwa orchestracji gameplayu;
- lokalny multiplayer 2 graczy, split screen pionowy, duża proceduralna mapa, 8 broni, niszczalny terrain, debug labs.

## Co nie wymaga osobnego crate’a w tym tasku

- `pickup_2d`: nie, mod entities + trigger flow wystarczą;
- `environment_2d`: nie na start, jeśli preset można złożyć z istniejących komponentów render/background/state;
- `projectile_2d`: nie na start, jeśli obecne emittery/projectile helpers da się rozszerzyć minimalnie;
- `weapons_2d`: nie, bronie i ammo pozostają mod-specific;
- `match_flow`: nie, całkowicie w modzie;
- `player_stats` / `hp` / `rounds`: nie, całkowicie w modzie.

## Definition of Done

- istnieje opis implementacyjny moda i fundamentów engine pod `terrain duel`;
- dokument jasno oddziela: co jest engine-generic, co jest mod-specific;
- niszczalność w engine jest opisana jako `destructible_surface_2d`, nie terrain-specific domain;
- pickupy są opisane jako scene entities + scripted collect behavior, bez wymuszonego crate `pickup_2d`;
- dokument wskazuje, które proponowane wcześniej crate’y są naprawdę konieczne, a które mają zostać mod-first do czasu realnego reuse.

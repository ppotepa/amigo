# TASK 017 Implementation Plan

## 0. Kontekst

Ten task opisuje plan dla moda `playground-2d-terrain-duel`.

Punkt wyjścia:

- engine ma już service-based runtime architecture;
- istnieją capability/pluginy pokrywające m.in. `motion_2d`, `physics_2d`, `particles_2d`;
- istnieją projectile emitters, particle bursts, collision rules i Rhai packages `std` / `arcade_2d`;
- celem nie jest zbudowanie gry-specyficznej logiki w engine, tylko dołożenie brakujących, generycznych domen.

Najważniejsza korekta względem wcześniejszego planu:

- nie każdy system potrzebuje osobnego crate’a w MVP;
- pickupy mają zostać zwykłymi encjami sceny z zachowaniem skryptowym po zebraniu;
- environment i projectile runtime mogą zacząć jako mod-first / thin-extension, jeśli nie ma jeszcze twardego dowodu na reuse.

---

## 1. Cel moda

`playground-2d-terrain-duel` ma być poligonem testowym dla:

- niszczalnych powierzchni/masek 2D użytych jako terrain;
- deformacji maski i geometrii przez eksplozje i bronie;
- dużego proceduralnego świata;
- lokalnego multiplayera 2 graczy;
- split screena;
- per-player UI;
- projectile + particle workflow;
- prostych pickupów;
- debugowych scen laboratoryjnych.

To nie ma być implementacja “Worms-specific engine code”.

Engine ma dostać tylko fundamenty ogólne:

- runtime-edytowalne destructible surfaces;
- viewport/split screen;
- query helpers, jeśli są naprawdę reużywalne.

Wszystko inne zostaje w modzie:

- HP;
- broń;
- ammo;
- rundy;
- zwycięstwo;
- pickup effects;
- match rules.

## 2. Decyzje architektoniczne

### 2.1. Wymagane nowe crate’y

Osobny crate jest uzasadniony dla:

```text
crates/2d/destructible-surface
crates/2d/viewport
```

Powód:

- to są domeny wyraźnie generyczne;
- mają własny model danych, runtime service, scene bridge i Rhai API;
- mają sens poza tym jednym modem.

### 2.2. Systemy, które nie wymagają osobnego crate’a na MVP

#### Pickupy

Nie robić od razu `crates/2d/pickup`.

Pickupy w MVP mają być:

- zwykłymi encjami sceny;
- z `Transform2D`, warstwą wizualną i colliderem/triggerem;
- oznaczane tagiem, nazwą albo lekkim komponentem danych z YAML;
- obsługiwane przez istniejący event/trigger flow i Rhai.

Przepływ:

```text
1. Player overlapuje pickup trigger.
2. Script sprawdza, czy pickup jest aktywny.
3. Script aplikuje effect do state gracza.
4. Pickup jest ukrywany lub dezaktywowany.
5. Jeśli ma respawn, skrypt planuje ponowną aktywację.
```

To jest wystarczające i lepiej pasuje do obecnego silnika niż dokładanie crate’a tylko dla prostego collectable flow.

#### Environment presets

Nie wymuszać od razu `crates/2d/environment`.

MVP może trzymać:

- YAML presetów w modzie;
- background layers jako dane sceny;
- tint/ambient/parallax przez istniejące komponenty render/state;
- wybór pory dnia przez state key i Rhai apply step.

Crate ma sens dopiero wtedy, gdy:

- kilka modów używa tych samych presetów;
- potrzebny jest wspólny runtime service;
- pojawi się realny authoring flow współdzielony przez wiele scen/modów.

#### Projectile runtime

Nie wymuszać od razu `crates/2d/projectile`.

Najpierw sprawdzić, czy MVP da się złożyć przez:

- obecny `motion_2d`;
- obecny `physics_2d`;
- istniejące projectile emitters;
- cienkie helpery skryptowe i ewentualnie małe, generyczne rozszerzenia runtime.

Osobny crate ma sens dopiero wtedy, gdy pojawi się wyraźnie większy, współdzielony model:

- lifetime;
- gravity;
- collision modes;
- impact policies;
- bounce/drill/explode jako dane runtime.

#### Weapons

Nie robić `weapons_2d` crate.

Weapon set, ammo rules, cooldowny i wybór slotów są grą-specyficzne i zostają w:

```text
mods/playground-2d-terrain-duel/data/weapons.yml
mods/playground-2d-terrain-duel/scripts/terrain_duel/weapons.rhai
```

## 4. `destructible_surface_2d`

To nie jest terrain-specific feature.

Engine feature ma reprezentować generyczną runtime-edytowalną powierzchnię 2D:

- authoring może pochodzić z polygonów, sprite, heightfield, procedural generatora albo YAML;
- runtime trzyma maskę solidności/alpha field;
- brush operations modyfikują maskę;
- dirty regions mówią rendererowi/collision, co trzeba odświeżyć;
- query API odpowiada na `solid_at`, `raycast`, `circle_cast` i podobne pytania.

Terrain duel używa tej powierzchni jako terenu. Inne mody mogą użyć tego samego systemu jako niszczalnych ścian, asteroid, lodu, osłon, barier, scorch masks albo edytorowych masek.

### 4.1. Nowy crate

Dodać:

```text
crates/2d/destructible-surface/
```

Capability:

```text
destructible_surface_2d
```

To jest obowiązkowa nowa domena engine.

### 4.2. Zakres crate’a

- `DestructibleSurface2d`;
- `SurfaceMask2d`;
- `SurfaceMaterial2d`;
- brush / damage commands;
- dirty regions;
- opcjonalny generator powierzchni;
- surface queries;
- surface collision MVP;
- scene bridge;
- Rhai bindings;
- testy jednostkowe.

### 4.3. MVP rendering/collision

Na MVP:

- render może być prostszy niż docelowy contour/mesh;
- collision może być mask-based, bez pełnego wciskania dynamicznego terrain do obecnego physics jako static collider mesh.

Najważniejsze jest stabilne API:

- `solid_at`;
- `raycast`;
- `subtract_circle`;
- `cut_capsule` albo równoważny brush;
- `dirty_regions`;
- `resolve_circle_vs_surface`.

### 4.4. Nazwy typów

Preferowane nazwy engine:

```text
DestructibleSurface2d
SurfaceMask2d
SurfaceBrush2d
SurfaceDamage2d
SurfaceDirtyRegion2d
SurfaceMaterial2d
SurfaceRaycastHit2d
SurfaceGenerator2d
```

Nazwy `Terrain*` mogą występować po stronie moda, danych albo scen testowych, kiedy opisują konkretne użycie:

```yaml
- type: DestructibleSurface2D
  id: main-terrain
  material: grass_soil
  generator:
    kind: mixed_terrain
```

## 6. Pickupy bez nowego crate’a

### 6.1. Model MVP

Pickup ma być po prostu encją sceny:

```text
entity
  transform
  sprite/vector
  trigger collider
  tags/data for kind
```

Dane konfiguracyjne:

```yaml
pickup_rules:
  - kind: health_pack
    count: 3
    respawn_seconds: 30.0
    allowed_zones: [surface, mid_caves]
    min_distance_between: 500.0
```

Same spawned pickup instances nadal są zwykłymi entities.

### 6.2. Runtime flow

Runtime ma używać istniejącego silnika, nie osobnego pickup service:

```text
1. Match scene albo script pyta destructible surface/terrain query o kandydatów spawn.
2. Script wybiera pozycje według rules.
3. Script spawnuje lub aktywuje pickup entities.
4. Trigger/collision event zgłasza overlap z graczem.
5. `pickups.rhai` sprawdza typ pickupu i aplikuje efekt.
6. Pickup jest hidden/disabled.
7. Respawn obsługiwany timerem w state/script.
```

### 6.3. Kiedy dopiero wydzielić crate

`pickup_2d` ma sens dopiero, jeśli pojawi się reuse w kilku modach i wspólny model:

- spawn rules;
- respawn timers;
- active/inactive lifecycle;
- collect events;
- surface-query placement helpers.

Bez tego będzie to przedwczesna abstrakcja.

## 8. Projectile/weapon workflow

### 8.1. Gameplay vs visuals

Broń nie może być “samymi particles”.

Model:

```text
Gameplay:
  projectile / hitscan / explosion / terrain damage / actor damage

Visual:
  muzzle / trail / impact / smoke / debris przez particles
```

### 8.2. Gdzie trzymać weapon logic

W modzie:

- `weapons.yml`;
- `weapons.rhai`;
- `explosions.rhai`;
- `projectiles.rhai`.

### 8.3. Co można dołożyć do istniejących crate’ów zamiast nowego

Dopuszczalne małe rozszerzenia:

- `motion_2d`: angle override, profile id, prostsze spawn helpers;
- `particles_2d`: `burst_at`, attached trail convenience, preset helpers;
- `physics_2d`: hook/event dla external impact integration.

Nie dokładać od razu pełnego `projectile_2d`, jeśli obecny workflow da się złożyć mniejszym kosztem.

## 9. Kolejność implementacji

### Phase 1: destructible surface core

```text
1. Dodać `crates/2d/destructible-surface`.
2. Zaimplementować maskę, circle subtraction, dirty region, `solid_at`.
3. Dodać scene bridge i podstawowe testy.
4. Dodać terrain-lab scene.
```

### Phase 2: generator + terrain-lab

```text
1. Dodać deterministic generator.
2. Dodać zones i spawn queries.
3. Dodać debug overlay dla mask/dirty/chunks.
```

### Phase 3: viewport + split screen

```text
1. Dodać `crates/2d/viewport`.
2. Dodać dwa viewporty, follow cameras i per-viewport HUD.
3. Dodać split-screen-lab.
```

### Phase 4: surface collision

```text
1. Dodać mask-based player/projectile vs destructible surface.
2. Dodać raycast i spawn point queries.
3. Spiąć movement gracza z surface collision używaną przez terrain.
```

### Phase 5: weapons MVP

```text
1. Zostać przy mod-first weapon orchestration.
2. Zaimplementować pistol, shotgun, rocket.
3. Podpiąć particles i terrain damage.
```

### Phase 6: match MVP

```text
1. Dodać menu, match, result.
2. Dodać 2 graczy, HP, cooldowny, ammo.
3. Dodać pickup entities + scripted collect/respawn.
```

### Phase 7: full arsenal + polish

```text
1. Dodać 8 broni.
2. Dodać environment presets z danych moda.
3. Dodać labs i wynik rundy.
```

## 10. Zakazy architektoniczne

Nie robić:

```text
- Worms/Liero-specific nazw w engine;
- HP/rund/ammo/broni konkretnej gry w engine;
- osobnego crate’a dla pickupów bez twardego reuse;
- osobnego crate’a dla environment tylko dla jednego moda;
- osobnego crate’a dla weapons;
- pełnego projectile crate’a, jeśli wystarczy cienkie rozszerzenie istniejących runtime’ów;
- particles jako jedynego źródła gameplay collision;
- global HUD-only bez viewport abstraction.
```

Robić:

```text
- destructible surface i viewport jako osobne capabilities;
- resztę mod-first, jeśli nie ma jeszcze realnego współdzielenia;
- YAML-first data;
- Rhai jako cienką orchestrację;
- deterministic seed;
- labs/debug scenes;
- dirty regions i chunking od początku.
```

## 11. Definition of Done

### Engine

```text
- istnieje capability `destructible_surface_2d`;
- destructible surface można zdefiniować w YAML;
```

```text
- istnieje `playground-2d-terrain-duel`;

- działa MVP broni projectile-heavy;
```

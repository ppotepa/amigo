# Amigo Editor Cache

## Architecture Decision

`amigo-editor` has a dedicated cache subsystem. Cache entries are keyed by stable `projectCacheId`, not by display name. Scene previews are cached as engine-generated slideshow manifests and frame files.

The current development cache root may remain under:

```txt
target/amigo-editor/cache
```

but the cache root is resolved through a provider so it can later move to platform-specific AppCache/AppData locations.

## Current Backend Shape

```txt
src-tauri/src/cache/
├─ mod.rs
├─ root.rs
└─ project_id.rs

src-tauri/src/preview/
├─ cache.rs
├─ hashing.rs
├─ renderer.rs
└─ slideshow.rs
```

The `preview` module generates and consumes preview artifacts. The `cache` module owns cache root resolution and project cache identity.

## Cache Root

Current resolution order:

```txt
1. AMIGO_EDITOR_CACHE_DIR
2. target/amigo-editor/cache
```

Future resolution should use Tauri/AppCache injection instead of hardcoded platform paths.

## Project Cache ID

The cache must distinguish:

```txt
displayName      - visible UI name
modId            - technical manifest id
projectCacheId   - stable cache identity
```

Current non-invasive resolution:

```txt
1. read <mod-root>/.amigo-editor/project-id.json if it exists
2. fallback to hash(canonical mod root path)
```

This avoids writing editor files into mods during discovery. A later explicit “adopt project” action may create `.amigo-editor/project-id.json`.

## Preview Cache Layout

```txt
<cache-root>/
└─ projects/
   └─ <project-cache-id>/
      └─ previews/
         └─ scenes/
            └─ <scene-id>/
               └─ <source-hash>/
                  ├─ preview.json
                  ├─ frame_000.png
                  ├─ frame_001.png
                  └─ frame_014.png
```

`modId` and display name may appear in manifests for diagnostics, but they are not the root cache key.

## Scene Preview Manifest

`preview.json` stores:

```txt
kind
version
projectCacheId
modIdAtGeneration
sceneLabelAtGeneration
sourceHash
rendererVersion
width
height
fps
frameCount
generatedAt
inputs
frames
```

The current slideshow is generated by the engine preview host:

```txt
frame_000.png -> capture_rgba8()
frame_001..  -> capture_next_frame()
```

## Invalidation

Current source hash includes:

```txt
engine-slideshow-preview-v1
width
height
fps
frameCount
mod.toml
scene.yml / scene.yaml
scene.rhai
```

Future inputs should include referenced assets and renderer/contract schema versions where available.

## Policy

Initial policy:

```txt
max age: unlimited
max size: none
cleanup: manual only
```

Do not automatically delete cache in early editor iterations; generated frames are useful for debugging preview failures.

## Fit With Editor Architecture

This matches the editor dependency direction:

```txt
engine contracts/runtime -> editor backend adapter -> frontend UI
```

The editor backend owns cache orchestration. The frontend only receives DTOs and display URLs. It does not scan cache folders or interpret cache layout.

## Later Commands

Planned command surface:

```txt
get_cache_info()
reveal_cache_folder()
clear_project_cache(projectCacheId)
clear_preview_cache(projectCacheId)
regenerate_all_scene_previews(modId)
```

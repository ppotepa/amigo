# TASK 014 Implementation Plan

## 1. UI model

W `crates/ui/core/src/model.rs`:

- `UiNode.style_class: Option<String>`
- `UiNodeKind::Slider { value: f32, min: f32, max: f32, step: f32 }`
- `UiNodeKind::Toggle { checked: bool, text: String, font: Option<AssetKey> }`
- `UiEvents.on_change: Option<UiEventBinding>`

Wymagane zachowanie:

- stare dokumenty bez `style_class` działają bez zmian;
- `Slider.value` jest mapowany do `0..1` przez `UiStateService`;
- `Toggle.checked` może być override'owany przez value `0.0/1.0` na tym samym path.

pub struct UiThemePalette {

Dodać `UiThemeService`:

```rust
register_theme(theme) -> bool

```

`UiPlugin` rejestruje `UiThemeService`.

Clear sceny czyści `UiThemeService`.

## 6. Resolve UI overlay

W `crates/apps/app/src/ui_runtime.rs`:

- dodać `UiThemeService` do resolvera;
- przy każdym node:
  - skopiować inline style;
  - jeśli aktywny theme i `style_class`, merge class style z inline style;
  - dopiero potem color override.

Ważne: `UiStyle` dziś nie ma informacji, czy `padding/gap/font_size/border_width` są ustawione inline czy domyślne. Minimalny pierwszy wariant może merge'ować tylko pola `Option` oraz zostawić numery inline jako ostateczne. Lepszy wariant: dodać optional numeric style document i runtime style, ale to większa zmiana. Przyjmujemy wariant pragmatyczny: theme nadaje przede wszystkim kolory/tła/border/font size, a layout w YAML zostaje inline.

## 8. Slider/toggle input

W `UiInputService`:

- dodać `active_drag_path: Option<String>`;
- input system na mouse down/release/down:
  - hit-testuje node;
  - jeśli node jest sliderem, ustawia drag;
  - drag wylicza value z pozycji X względem rect;
  - zapisuje `UiStateService.set_value(path, value)`;
  - publikuje `on_change`.

Toggle:

- click/release na toggle odwraca value;
- zapisuje `0.0/1.0`;
- publikuje `on_change`.

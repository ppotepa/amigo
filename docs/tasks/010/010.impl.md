# TASK 010 - Asteroids Playground Refactor Through Agnostic Engine APIs

## Status

Planned.

## Type

Architecture Refactor / Engine Capability Extraction / Scripting Simplification

## Priority

High

## Why This Task Exists

`playground-2d-asteroids` działa, ale obecny `scene.rhai` pełni rolę mini-silnika:

1. trzyma ręcznie stan ruchu statku i asteroid,
2. zarządza ręcznie pulą pocisków i cooldownami,
3. robi ręczne skanowanie kolizji i kandydatów,
4. ręcznie odświeża HUD i timery,
5. ukrywa encje przez teleportację poza ekran.

To stoi w sprzeczności z celem: Asteroids ma wymuszać dopracowanie generycznych feature'ów engine, a nie produkować Asteroids-specific runtime.

## Core Rule

Nowe mechaniki muszą być:

1. agnostyczne względem gry,
2. plugin-first,
3. SRP,
4. testowalne bez runtime tam, gdzie to możliwe,
5. YAML-first, Rhai-thin,
6. bez offscreen-hacków do ukrywania,
7. dostępne zarówno deklaratywnie w YAML, jak i imperatywnie przez API runtime tam, gdzie ma to sens.

## Architectural Guardrails

### 1. Entity State Must Be Multi-Flag, Not A Single `active`

Nie wprowadzać zbyt wąskiego modelu `visible + active`, jeżeli miałby mieszać kilka odpowiedzialności.

Docelowy kierunek:

1. `visible`
2. `simulation_enabled`
3. `collision_enabled`

Opcjonalnie można zostawić user-facing aliasy `show/hide`, `enable/disable`, ale storage powinien pozwalać odróżnić:

1. encję niewidzialną, ale nadal aktywną logicznie,
2. encję widzialną, ale bez kolizji,
3. encję całkowicie wyłączoną z update loop.

### 2. One Shared Selector Model Across Systems

Nie projektować osobnych selectorów dla physics, pools, activation i query APIs.

Powinno istnieć jedno wspólne pojęcie:

```rust
pub enum EntitySelector {
    Entity(String),
    Tag(String),
    Group(String),
    Pool(String),
}
```

To samo pojęcie ma być używane wszędzie, gdzie użytkownik wskazuje zbiory encji.

### 3. Add Generic `properties` Early

Asteroids potrzebuje pól takich jak `score_value`, `split_set`, `wave`, `audio cue`, `shape seed`.

Nie należy rozwiązywać tego przez mnożenie specjalnych pól per feature. Potrzebne jest generyczne metadata storage:

1. entity-level `properties`,
2. scene-level `properties` jeśli potrzebne,
3. dostęp z Rhai i runtime.

### 4. State And Timers Need Scope-Aware Design

Pierwsze wdrożenie może być scene-scoped, ale API nie powinno zamykać drogi do:

1. `scene` scope,
2. `mod/session` scope,
3. ewentualnie `save/profile` scope w przyszłości.

To dotyczy zarówno `SceneStateService`, jak i timerów.

### 5. UI Bindings Must Be Extensible Beyond Text

`text_bind` to dobry start, ale schema ma zostawić miejsce na:

1. `text_bind`,
2. `visible_bind`,
3. `enabled_bind`,
4. `value_bind`.

Nie trzeba wszystkiego implementować w pierwszym batchu, ale model nie powinien blokować tego kierunku.

### 6. Avoid Crate Explosion

Nie mnożyć crate'ów tylko po to, żeby każdy feature miał własny katalog.

Dozwolone są dwa warianty:

1. nowy crate, jeśli capability jest wyraźnie osobną domeną,
2. nowy moduł w istniejącym crate, jeśli granica domenowa już tam naturalnie pasuje.

Priorytetem jest czytelna granica capability, nie maksymalna liczba crate'ów.

## Desired End State

Po Task 010:

1. `scene.rhai` opisuje reguły gry, nie mechanikę silnika,
2. motion/bounds/pool/timers/state/collision-events/UI-binding/procedural-vector są reusable API engine,
3. scena deklaruje większość konfiguracji w `scene.yml`,
4. Asteroids używa tych API bez customowych usług typu `AsteroidsService`,
5. engine ma wspólny model selectorów, właściwości encji i wieloflagowy lifecycle encji,
6. nowe capability nadają się do wykorzystania nie tylko przez Asteroids, ale też przez inne gry arcade 2D, top-down i twin-stick.

## Workstream A - Entity Lifecycle Flags

### Goal

Zastąpić ukrywanie encji przez teleport (`x/y = 10000`) jawnym lifecycle encji.

### Actions

1. Dodać do `amigo_scene` stan encji z rozróżnieniem:
   - `visible`
   - `simulation_enabled`
   - `collision_enabled`
2. Rozszerzyć Rhai:
   - `world.entities.hide/show`
   - `world.entities.enable/disable`
   - jeśli potrzebne: `set_collision_enabled`
   - analogiczne metody na `EntityRef`
3. Render extraction ignoruje `visible == false`.
4. Physics/query/runtime systems respektują flagi symulacji i kolizji.

### Acceptance

1. `hide()` nie modyfikuje transformu.
2. Skrypt Asteroids nie używa `offscreen = 10000.0` jako stanu.
3. Feature pozwala rozróżnić niewidzialne obiekty, wyłączoną symulację i wyłączone kolizje.

## Workstream B - Entity Metadata: Tags, Groups, Properties

### Goal

Dostarczyć wspólny model opisu encji dla query, scoringu, podziału fal i reguł.

### Actions

1. Wykorzystać istniejące `tags` na encji.
2. Dodać `groups`.
3. Dodać generyczne `properties`.
4. Rozszerzyć query API:
   - `by_tag`
   - `by_group`
   - `active_by_tag`
   - `has_tag`
   - `property_*` accessors, jeśli potrzebne po stronie Rhai

### Acceptance

1. Asteroids nie trzyma ręcznie list asteroid po nazwach.
2. Score, split i podobne dane nie wymagają osobnych custom fields w silniku.

## Workstream C - Shared Selector Model

### Goal

Ujednolicić sposób wskazywania encji i zbiorów encji.

### Actions

1. Zdefiniować wspólny `EntitySelector`.
2. Użyć go w:
   - physics query,
   - collision event rules,
   - activation sets,
   - pool helpers, jeżeli to pasuje do modelu
3. Utrzymać kompatybilność prostych helperów entity-name based tam, gdzie to poprawia ergonomię.

### Acceptance

1. Użytkownik nie musi uczyć się kilku różnych selector DSL-i.
2. Query i event rules są spójne semantycznie.

## Workstream D - Input Convenience API

### Goal

Usunąć ręczne OR-y klawiszy z Rhai.

### Actions

1. Dodać do `input` bindings:
   - `axis(...)`
   - `any_down(...)`
   - `any_pressed(...)`
2. Zachować kompatybilność `down/pressed`.
3. Przygotować drogę do YAML input map (`actions`, `axes`) bez wiązania taska z pełnym remap systemem.

### Acceptance

1. Asteroids nie ma warunków typu `down("A") || down("ArrowLeft")`.
2. Są testy w `crates/scripting/rhai`.

## Workstream E - UI Batch API

### Goal

Zredukować wielokrotne `set_text(...)` do jednego wywołania tam, gdzie ręczne UI update jeszcze pozostaje.

### Actions

1. Dodać `world.ui.set_many(map)` w `UiApi`.
2. Zapewnić enqueue przez pojedynczy Rhai call.
3. Dodać helper `sync_hud()` oparty o `set_many` jako stan przejściowy przed pełnym bindingiem UI.

### Acceptance

1. Działa batch update dla score/lives/wave/status/debug.
2. Test w `crates/scripting/rhai`.

## Workstream F - State And Timers

### Goal

Usunąć ręczne odejmowanie cooldownów/timerów i przygotować reusable store stanu.

### Actions

1. Wprowadzić `SceneStateService` z typami:
   - `int`
   - `float`
   - `bool`
   - `string`
2. Zostawić miejsce na scope-aware design:
   - scene
   - session/mod
3. Wprowadzić timer service + plugin.
4. API Rhai:
   - `state.set_*`
   - `state.get_*`
   - `state.add_*`
   - `timers.start`
   - `timers.ready`
   - `timers.active`
   - `timers.after`

### Acceptance

1. Asteroids nie robi ręcznie `cooldown -= dt`.
2. Rhai nie trzyma HUD textów jako osobnych globali stanu technicznego.
3. Testy: start/tick/ready/reset/set/get/add.

## Workstream G - Runtime 2D: `Velocity2D`, `Bounds2D`, `FreeflightMotion2D`

### Goal

Przenieść ruch statku/asteroid/pocisków z Rhai do systemów runtime.

### Actions

1. Dodać `Velocity2D` komponent + system.
2. Dodać `Bounds2D` z behavior:
   - `bounce`
   - `wrap`
   - `despawn`
   - `hide`
   - `clamp`
3. Dodać `FreeflightMotion2D`:
   - profile
   - intent
   - state
   - system integrujący transform
4. Zaprojektować `FreeflightMotion2D` jako intent-driven capability, nie jako jednorazowe imperative teleport API.

### Acceptance

1. Skrypt nie trzyma `ship_vx/vy/angle/angular_velocity`.
2. Skrypt nie implementuje `next_ship_motion`, `next_bullet_state`, `step_asteroid_position`.
3. Unit tests dla modeli i systemów.
4. Runtime może zasilać intent zarówno z Rhai, jak i z innych źródeł.

## Workstream H - Projectile Pool + Lifetime

### Goal

Usunąć ręczne zarządzanie bullet slots.

### Actions

1. Dodać generyczny pool encji.
2. Dodać `ProjectileEmitter2D`:
   - `fire_from`
   - cooldown
   - speed
   - inherit velocity
   - spawn offset
3. Dodać `Lifetime`:
   - `hide`
   - `disable`
   - `despawn`
   - `return-to-pool`

### Acceptance

1. Asteroids nie zna `bullet-01/02/03` w logice strzału.
2. Pool używa lifecycle flags, nie teleport-hide.
3. Testy: acquire/release/reuse/no-slot/lifetime-expire.

## Workstream I - Physics Queries And Collision Events

### Goal

Zastąpić ręczne listy kandydatów i skanowanie kolizji event-driven API.

### Actions

1. Rozszerzyć physics query o wspólne selektory:
   - entity
   - tag
   - group
   - pool
2. Dodać `CollisionEventRule2D` z `once_per_overlap`.
3. Dodać payload zawierający minimum:
   - source entity
   - target entity
   - rule id lub topic context, jeśli potrzebne

### Acceptance

1. Asteroids nie używa ręcznego `first_overlap_index(..., asteroid_hit_candidates)`.
2. Eventy niosą obie encje.
3. Brak spamowania eventu przy utrzymanym overlapie.

## Workstream J - UI Bindings

### Goal

Przenieść HUD update na deklaratywne powiązania ze stanem.

### Actions

1. Dodać `UiDocument` bindings:
   - `text_bind`
2. Zaprojektować schema tak, żeby można było dodać później:
   - `visible_bind`
   - `enabled_bind`
   - `value_bind`
3. Dodać system `ui_state_binding` aktualizujący `UiStateService` ze stanu.

### Acceptance

1. Rhai ustawia tylko wartości stanu (`world.state.set_*`).
2. UI odświeża się deklaratywnie.
3. Asteroids nie trzyma `hud_*_text` jako osobnych globali.

## Workstream K - Procedural Vector Generation

### Goal

Usunąć ręczne generowanie punktów asteroid z Rhai.

### Actions

1. Dodać procedural generator API (`seeded`, deterministic).
2. Zamiast gameplay-specific nazwy użyć neutralnego generatora, np. `radial-jitter-polygon`.
3. Podpiąć YAML i/lub Rhai helper do generowania shape.

### Acceptance

1. `generate_asteroid_points` znika ze skryptu.
2. Test deterministyczny dla seeda.
3. Walidacja parametrów (`vertices >= 3`, clamp jitter).

## Workstream L - Activation Sets

### Goal

Zamiast ręcznego resetu encji w skrypcie użyć deklaratywnego activation preset.

### Actions

1. Dodać `ActivationSet` i `ActivationEntry`.
2. Wspierać:
   - lifecycle flags
   - transform
   - velocity
   - angular velocity
   - opcjonalny seed/properties override
3. API Rhai:
   - `world.activation.apply(id)`
   - `world.activation.clear_group(...)`

### Acceptance

1. Start fal nie ustawia ręcznie kilkunastu encji przez skrypt.
2. Mechanizm pozostaje reusable dla innych modów.

## Workstream M - Audio Cue Layer

### Goal

Przenieść throttling cue (np. thrust) z Rhai do audio API.

### Actions

1. Dodać scene-local `audio_cues` (alias + opcjonalny `min_interval`).
2. Dodać `world.audio.cue("name")`.

### Acceptance

1. Asteroids nie trzyma `thrust_audio_cooldown`.
2. Test `min_interval`.

## Workstream N - Scene Schema + Hydration + Packaging Boundary

### Goal

Dodać pełne wsparcie YAML dla nowych komponentów i nie przesadzić z atomizacją crate'ów.

### Actions

1. Rozszerzyć schema (`document.rs`) i scene commands.
2. Dodać dedykowane handlery SRP w app runtime.
3. Dodać scene events `*Queued` i logi dev console.
4. Dla każdego capability świadomie zdecydować:
   - osobny crate
   - czy moduł w istniejącym crate

### Acceptance

1. Każdy nowy component ma command + handler + test hydration.
2. Scena Asteroids jest deklaratywna tam, gdzie to możliwe.
3. Repo nie dostaje niepotrzebnego crate explosion.

## Workstream O - Asteroids Refactor (`scene.yml` + `scene.rhai`)

### Goal

Przepisać Asteroids na nowy contract engine.

### Actions

1. `scene.yml`:
   - usunąć offscreen spawn hack
   - dodać lifecycle flags
   - dodać tags/groups/properties
   - dodać pools
   - dodać collision event rules
   - dodać motion/bounds/velocity/lifetime/emitter
   - dodać UI bindings
2. `scene.rhai`:
   - zostawić reguły gry (`wave`, `hit`, `respawn`, `score`, `game over`)
   - usunąć mini-engine logikę ruchu/kolizji/pool/timer/HUD spam

### Acceptance

1. Skrypt jest cienką warstwą rules.
2. Nie zawiera ręcznego engine loop dla fizyki/kolizji/pocisków.

## Validation Matrix

Po każdym większym batchu uruchamiać:

1. `cargo test -p amigo-scene`
2. `cargo test -p amigo-scripting-rhai`
3. `cargo test -p amigo-app`
4. testy crate'ów lub modułów nowych feature'ów (`motion`, `bounds`, `projectile`, `timers`, `state`, `vector procedural`)

Dodatkowo:

1. runtime bootstrap test dla `playground-2d-asteroids/vector-preview`
2. smoke test: fire -> bullet active -> collision event -> score update
3. grep, że Rhai nie zawiera starych ręcznych struktur stanu
4. testy, że imperative API i YAML path dają ten sam efekt capability tam, gdzie wspieramy obie ścieżki

## Definition of Done

Task 010 jest zakończony, gdy:

1. `scene.rhai` nie trzyma ręcznych tablic bullet state i ship kinematics,
2. `scene.rhai` nie używa offscreen hide (`10000`),
3. `scene.rhai` nie robi ręcznego overlap scan po kandydatach asteroid,
4. `scene.rhai` nie generuje ręcznie punktów asteroid,
5. `scene.yml` deklaruje lifecycle flags, tags/groups/properties, pool, bounds, motion, collision events, UI bindings,
6. engine udostępnia nowe reusable API bez nazw Asteroids,
7. nowe capability mają wspólny model selectorów i rozsądny packaging boundary,
8. wszystkie nowe feature'y mają testy jednostkowe + runtime smoke tests.

## Guidance For The Next Agent

1. Nie budować `Asteroids*` services ani systemów.
2. Najpierw dostarczać API generyczne, potem przepinać mod.
3. Utrzymać strict SRP i modułowe bindingi Rhai.
4. Nie wprowadzać zbyt wąskich modeli, które za chwilę wymuszą kolejne breaking zmiany (`active`, `text_bind`, custom selectors).
5. Traktować refaktor jako migration to engine capability, nie jako quick patch sceny.

# TASK 010 - Asteroids Playground Refactor Through Agnostic Engine APIs

## Status

Planned.

## Type

Architecture Refactor / Engine Capability Extraction / Scripting Simplification

## Priority

High

## Why This Task Exists

`playground-2d-asteroids` działa, ale obecny `scene.rhai` pełni rolę mini-silnika:

1. trzyma ręcznie stan ruchu statku i asteroid,
2. zarządza ręcznie pulą pocisków i cooldownami,
3. robi ręczne skanowanie kolizji i kandydatów,
4. ręcznie odświeża HUD i timery,
5. ukrywa encje przez teleportację poza ekran.

To stoi w sprzeczności z celem: Asteroids ma wymuszać dopracowanie generycznych feature’ów engine, a nie produkować Asteroids-specific runtime.

## Core Rule

Nowe mechaniki muszą być:

1. agnostyczne względem gry,
2. plugin-first,
3. SRP,
4. testowalne bez runtime tam, gdzie to możliwe,
5. YAML-first, Rhai-thin,
6. bez offscreen-hacków do ukrywania.

## Desired End State

Po Task 010:

1. `scene.rhai` opisuje reguły gry, nie mechanikę silnika,
2. motion/bounds/pool/timers/state/collision-events/UI-binding/procedural-vector są reusable API engine,
3. scena deklaruje większość konfiguracji w `scene.yml`,
4. Asteroids używa tych API bez customowych usług typu `AsteroidsService`.

## Workstream A - Entity State (`visible` / `active`)

### Goal

Zastąpić ukrywanie encji przez teleport (`x/y = 10000`) jawnych stanem encji.

### Actions

1. Dodać `visible` i `active` do stanu encji w `amigo_scene`.
2. Rozszerzyć Rhai:
   - `world.entities.hide/show/disable/enable`
   - analogiczne metody na `EntityRef`.
3. Render extraction ignoruje `visible == false`.
4. Query/physics ignoruje `active == false` (lub ma parametr `include_inactive`).

### Acceptance

1. `hide()` nie modyfikuje transformu.
2. Skrypt Asteroids nie używa `offscreen = 10000.0` jako stanu.

## Workstream B - Input Convenience API

### Goal

Usunąć ręczne OR-y klawiszy z Rhai.

### Actions

1. Dodać do `input` bindings:
   - `axis(...)`
   - `any_down(...)`
   - `any_pressed(...)`
2. Zachować kompatybilność `down/pressed`.
3. Opcjonalnie przygotować drogę do YAML input map (`actions`, `axes`).

### Acceptance

1. Asteroids nie ma warunków typu `down("A") || down("ArrowLeft")`.
2. Są testy w `crates/scripting/rhai`.

## Workstream C - UI Batch API

### Goal

Zredukować wielokrotne `set_text(...)` do jednego wywołania.

### Actions

1. Dodać `world.ui.set_many(map)` w `UiApi`.
2. Zapewnić enqueue przez pojedynczy Rhai call.
3. Dodać helper `sync_hud()` oparty o `set_many`.

### Acceptance

1. Działa batch update dla score/lives/wave/status/debug.
2. Test w `crates/scripting/rhai`.

## Workstream D - Timers API

### Goal

Usunąć ręczne odejmowanie cooldownów/timerów w Rhai.

### Actions

1. Wprowadzić timer service (`scene-scoped`) + plugin.
2. API Rhai:
   - `start`
   - `ready`
   - `active`
   - `after`
3. Czyścić timery przy reloadzie sceny.

### Acceptance

1. Asteroids nie robi ręcznie `cooldown -= dt`.
2. Testy: start/tick/ready/reset.

## Workstream E - Runtime 2D: `Velocity2D`, `Bounds2D`, `FreeflightMotion2D`

### Goal

Przenieść ruch statku/asteroid/pocisków z Rhai do systemów runtime.

### Actions

1. Dodać `Velocity2D` komponent + system.
2. Dodać `Bounds2D` z behavior:
   - `bounce`
   - `wrap`
   - `despawn`
   - `hide`
   - `clamp`
3. Dodać `FreeflightMotion2D`:
   - profile
   - intent
   - state
   - system integrujący transform.

### Acceptance

1. Skrypt nie trzyma `ship_vx/vy/angle/angular_velocity`.
2. Skrypt nie implementuje `next_ship_motion`, `next_bullet_state`, `step_asteroid_position`.
3. Unit tests dla modeli i systemów.

## Workstream F - Projectile Pool + Lifetime

### Goal

Usunąć ręczne zarządzanie bullet slots.

### Actions

1. Dodać generyczny pool encji.
2. Dodać `ProjectileEmitter2D` (`fire_from`, cooldown, speed, inherit velocity, offset).
3. Dodać `Lifetime` (`hide/disable/despawn/return-to-pool`).

### Acceptance

1. Asteroids nie zna `bullet-01/02/03` w logice strzału.
2. Pool używa `active/visible`, nie teleport-hide.
3. Testy: acquire/release/reuse/no-slot/lifetime-expire.

## Workstream G - Tags, Queries, Collision Events

### Goal

Zastąpić ręczne listy kandydatów i skanowanie kolizji event-driven API.

### Actions

1. Dodać tagi/grupy encji w scenie.
2. Dodać query API:
   - `by_tag`
   - `by_group`
   - `active_by_tag`
3. Rozszerzyć physics query o selektory: entity/tag/group/pool.
4. Dodać `CollisionEventRule2D` z `once_per_overlap`.

### Acceptance

1. Asteroids nie używa ręcznego `first_overlap_index(..., asteroid_hit_candidates)`.
2. Eventy niosą obie encje.
3. Brak spamowania eventu przy utrzymanym overlapie.

## Workstream H - Scene State + UI Bindings

### Goal

Przenieść gameplay state i powiązania HUD do serwisów engine.

### Actions

1. Dodać `SceneStateService` (`int/float/bool/string`).
2. Dodać `UiDocument` text bindings (`text_bind`).
3. Dodać system `ui_state_binding` aktualizujący `UiStateService` ze stanu.

### Acceptance

1. Rhai ustawia tylko wartości stanu (`world.state.set_*`).
2. UI odświeża się deklaratywnie.
3. Asteroids nie trzyma `hud_*_text` jako osobnych globali.

## Workstream I - Procedural Vector Generation

### Goal

Usunąć ręczne generowanie punktów asteroid z Rhai.

### Actions

1. Dodać procedural generator API (`seeded`, deterministic).
2. Dodać generator polygonu typu `rock-polygon` (agnostyczna nazwa).
3. Podpiąć YAML i/lub Rhai helper do generowania shape.

### Acceptance

1. `generate_asteroid_points` znika ze skryptu.
2. Test deterministyczny dla seeda.
3. Walidacja parametrów (`vertices >= 3`, clamp jitter).

## Workstream J - Activation Sets

### Goal

Zamiast ręcznego resetu encji w skrypcie użyć deklaratywnego activation preset.

### Actions

1. Dodać `ActivationSet` i `ActivationEntry`.
2. Wspierać `active`, `visible`, `transform`, `velocity`, `angular`, opcjonalny seed.
3. API Rhai:
   - `world.activation.apply(id)`
   - `world.activation.clear_group(...)`

### Acceptance

1. Start fal nie ustawia ręcznie kilkunastu encji przez skrypt.
2. Mechanizm pozostaje reusable dla innych modów.

## Workstream K - Audio Cue Layer

### Goal

Przenieść throttling cue (np. thrust) z Rhai do audio API.

### Actions

1. Dodać scene-local `audio_cues` (alias + opcjonalny `min_interval`).
2. Dodać `world.audio.cue("name")`.

### Acceptance

1. Asteroids nie trzyma `thrust_audio_cooldown`.
2. Test `min_interval`.

## Workstream L - Scene Schema + Hydration + Handlers

### Goal

Dodać pełne wsparcie YAML dla nowych komponentów i konfiguracji.

### Actions

1. Rozszerzyć schema (`document.rs`) i scene commands.
2. Dodać dedykowane handlery SRP w app runtime.
3. Dodać scene events `*Queued` i logi dev console.

### Acceptance

1. Każdy nowy component ma command + handler + test hydration.
2. Scena Asteroids jest deklaratywna tam, gdzie to możliwe.

## Workstream M - Asteroids Refactor (`scene.yml` + `scene.rhai`)

### Goal

Przepisać Asteroids na nowy contract engine.

### Actions

1. `scene.yml`:
   - usunąć offscreen spawn hack
   - dodać `active/visible`
   - dodać tags
   - dodać pools
   - dodać collision event rules
   - dodać motion/bounds/velocity/lifetime/emitter
   - dodać UI bindings
2. `scene.rhai`:
   - zostawić reguły gry (`wave`, `hit`, `respawn`, `score`, `game over`)
   - usunąć mini-engine logikę ruchu/kolizji/pool/timer/HUD spam.

### Acceptance

1. Skrypt jest cienką warstwą rules.
2. Nie zawiera ręcznego engine loop dla fizyki/kolizji/pocisków.

## Validation Matrix

Po każdym większym batchu uruchamiać:

1. `cargo test -p amigo-scene`
2. `cargo test -p amigo-scripting-rhai`
3. `cargo test -p amigo-app`
4. testy crate’ów nowych feature’ów (`motion`, `bounds`, `projectile`, `timers`, `state`, `vector procedural`).

Dodatkowo:

1. runtime bootstrap test dla `playground-2d-asteroids/vector-preview`,
2. smoke test: fire -> bullet active -> collision event -> score update,
3. grep, że Rhai nie zawiera starych ręcznych struktur stanu.

## Definition of Done

Task 010 jest zakończony, gdy:

1. `scene.rhai` nie trzyma ręcznych tablic bullet state i ship kinematics,
2. `scene.rhai` nie używa offscreen hide (`10000`),
3. `scene.rhai` nie robi ręcznego overlap scan po kandydat ach asteroid,
4. `scene.rhai` nie generuje ręcznie punktów asteroid,
5. `scene.yml` deklaruje tagi, pool, bounds, motion, collision events, UI bindings,
6. engine udostępnia nowe reusable API bez nazw Asteroids,
7. wszystkie nowe feature’y mają testy jednostkowe + runtime smoke tests.

## Guidance For The Next Agent

1. Nie budować `Asteroids*` services ani systemów.
2. Najpierw dostarczać API generyczne, potem przepinać mod.
3. Utrzymać strict SRP i modułowe bindingi Rhai.
4. Traktować refaktor jako migration to engine capability, nie jako quick patch sceny.

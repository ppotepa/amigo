# TASK 008 - Trait-First / Plugin-First / Game-Agnostic Refactor

## Status

Completed for architectural acceptance. Remaining legacy package/plugin rename work is follow-up, not a blocker for Task 008.

## Type

Architecture / Runtime Composition / Scripting / Rendering Decoupling

## Priority

High

## Goal

Przebudować Amigo tak, aby:

1. engine był neutralny gatunkowo (bez `platformer` jako core feature),
2. nowe capability dało się dodać pluginem bez edycji `crates/apps/app/src/lib.rs`,
3. boilerplate gameplay (sidescroller itp.) żył poza core engine.

## Guardrails

1. migration compatibility-first, bez big-bang rewrite,
2. każdy etap ma przechodzące testy i działający bootstrap,
3. najpierw mechaniczny podział i registry, potem rename/cleanup legacy.

## Current Baseline (validated)

1. `app` jest już częściowo rozdzielony, ale nadal pełni rolę centralnego orchestratora,
2. `SceneCommand` i scripting command path nadal są silnie domenowo sprzężone,
3. legacy motion identity nadal istnieje głównie na poziomie plugin/package naming i compatibility alias exports, nie w aktywnym publicznym API runtime,
4. renderer WGPU nadal pobiera część domen bezpośrednio.

## Progress Update

1. `RuntimeContext` wrappers (`required<T>`, `optional<T>`) są już używane w `app`.
2. `PluginBundle` zostało dodane do runtime (`RuntimeBuilder::with_bundle(...)`).
3. bootstrap `amigo-app` został przepięty z długiego łańcucha `with_plugin(...)` na bundle composition z zachowaną kolejnością pluginów.
4. `SceneCommand` routing działa przez registry + per-domain handlers (bez centralnego matcha po wszystkich wariantach).
5. `ScriptCommand` routing działa przez registry i został rozbity na moduły handlerów per namespace (`scene`, `render`, `asset`, `audio`, `ui`, `debug`, `dev-shell`).
6. `process_ui_input` nie siedzi już inline w `InteractiveRuntimeHostHandler`; działa przez `systems/ui_input.rs`.
7. `amigo-runtime` ma już minimalny generic scheduler foundation: `RuntimeSystem`, `SystemPhase`, `SystemRegistry`.
8. `host_runtime` używa runtime `SystemRegistry` + `SystemPhase` i nie ma już własnego `PhaseRunner`; helper systems `script_update` i `scene_transition` zostały wydzielone z hosta.
9. `ui_input` działa już jako normalny runtime system (`PreUpdate`) i korzysta z runtime service `UiInputViewportState` zamiast lokalnej konfiguracji hosta.
10. `amigo-app` został przepięty na `Motion*` surface w aktywnej ścieżce bootstrap/scene runtime/system; legacy `Platformer*` nadal żyje głównie w domain crate i compatibility enum variants.
11. `amigo-scene` ma motion-first helpers dla commands/events, a parser YAML utrzymuje `PlatformerController2D` tylko jako alias kompatybilności.
12. `amigo-scripting-rhai` ma `new_with_motion(...)`, `new_with_motion_and_vector(...)` i canonical `world.motion`; `world.platformer` zostało usunięte z aktywnego publicznego API.
13. `render-api` ma pierwszy packet seam: `RenderFramePacket<TOverlay>`, a app-side UI overlay extraction przechodzi przez packet zamiast inline listy w host redraw.
14. `render-api` ma już generic `RenderExtractor<TContext, TOverlay>` i `RenderExtractorRegistry<TContext, TOverlay>`; `ui_runtime` używa tego jako pierwszego wspólnego extractor contractu.
15. `render-api` ma też `RenderFrameExtractor<TContext, TPacket>` i `RenderFrameExtractorRegistry<TContext, TPacket>`; app-side `render_runtime` wykorzystuje to do wspólnej ekstrakcji `TileMap2D`, `Sprite2D`, `Text2D`, `Vector2D`, `Mesh3D`, `Material3D`, `Text3D` i UI overlay.
16. `amigo-app` nie buduje już centralnie jednego `SystemRegistry`; rejestracja runtime systems została rozbita na mniejsze pluginy (`UiInputRuntimeSystemPlugin`, `ScriptUpdateRuntimeSystemPlugin`, `World2dRuntimeSystemsPlugin`, `SceneTransitionRuntimeSystemPlugin`, `AudioRuntimeSystemPlugin`).
17. `amigo-2d-platformer` zostało odwrócone wewnętrznie: canonical source definitions są już `Motion*`, a `Platformer*` to aliasy kompatybilności.
18. runtime po `MOTION_2D_PLUGIN` publikuje już tylko `Motion2dDomainInfo`; `PlatformerDomainInfo` nie jest już rejestrowane jako aktywna capability runtime.
19. capability reporting po stronie app traktuje `motion_2d` jako canonical; legacy plugin label jest raportowany jako `motion_2d via amigo-2d-platformer`.
20. diagnostics/summary nie mapują już ręcznie legacy nazwy pluginu motion; canonical runtime report label pochodzi z helpera domenowego `amigo-2d-platformer`.
21. `amigo-app` nie trzyma już ręcznie literalnego legacy motion label w diagnostics ani w bootstrap test expectations; używa canonical constant z domeny motion.
22. `engine/scene` nie ma już aktywnych enum variants `QueuePlatformerController2d` / `PlatformerControllerQueued`; canonical surface jest już czysto motion-first, a parser YAML utrzymuje tylko alias dokumentu.
23. `amigo-scripting-rhai` nie ma już compatibility constructor path `new_with_platformer(...)`; canonical construction idzie przez `new_with_motion(...)`.
24. `world.platformer` zostało usunięte z aktywnego publicznego API Rhai; `world.motion` jest jedyną aktywną ścieżką dla tej domeny.
25. `amigo-app` redraw path odbudowuje lokalne snapshot services z extractor packetu także dla `Mesh3D`, `Material3D` i `Text3D`; runtime `host_runtime` nie przekazuje już bezpośrednio rendererowi runtime scene services dla aktywnego 2D/3D content path.

## Latest Audit Signals

1. Największy pozostały legacy debt dla `platformer -> motion` siedzi w:
   `crates/2d/platformer`, `crates/scripting/rhai`, `crates/engine/scene`.
2. Najniższy-risk pierwszy slice dla render decoupling był UI overlay packet seam; app-side extractor registry nad tym pakietem obejmuje już cały obecny aktywny 2D/3D world path (`TileMap2D`, `Sprite2D`, `Text2D`, `Vector2D`, `Mesh3D`, `Material3D`, `Text3D`), więc następny sensowny slice to backend-side packet ownership w `render-wgpu`.

## Delivery Streams

### Stream A - Composition Root and Contexts

1. domknąć mechanical split `app` (czytelne granice bootstrap/orchestration/systems/assets/scene_runtime),
2. wprowadzić `RuntimeContext` (`required<T>`, `optional<T>`) i użyć go w dispatcherach,
3. wprowadzić `PluginBundle` i przenieść długą listę `.with_plugin(...)` do bundle layers.

### Stream B - Scene and Script Dispatch

1. dokończyć `SceneCommandHandlerRegistry` (pełna eliminacja centralnego domenowego `match` w app),
2. dokończyć `ScriptCommandHandlerRegistry` (namespace/command routed handlers),
3. przygotować compatibility bridge pod typed command bus.

### Stream C - Motion Neutralization

1. finalizować neutralizację `platformer` -> `motion`,
2. zostawić kompatybilność przez ograniczony czas (`world.platformer` alias),
3. usunąć `platformer_2d` z diagnostics/capabilities po zamknięciu migracji contentu.

### Stream D - Systems and Scheduling

1. wdrożyć `RuntimeSystem` + `SystemPhase` + `SystemRegistry`,
2. przenieść tick path z funkcji app do systemów plugin-owned,
3. host loop oparty o schedule phases.

### Stream E - Assets Pipeline Extensibility

1. `AssetProcessorRegistry` (`AssetLoader` / `AssetPreparer` / `PreparedAssetProcessor`),
2. wyprowadzić sprite/tile/audio metadata sync z `app` do plugin processors,
3. zachować hot reload behavior i testy bootstrap summary.

### Stream F - Scripting Architecture

1. `ScriptImportResolver` + moduły (`engine/mod/scene` origins),
2. plugin-owned Rhai bindings (`ScriptApiProvider`, `ScriptLibraryProvider`),
3. stopniowe odejście od monolitycznego `register_world_api`.

### Stream G - Scene Hydration Extensibility

1. `SceneComponentHydratorRegistry`,
2. każdy feature rejestruje hydrator komponentu YAML,
3. `scene core` bez domenowych branchy.

### Stream H - Render Decoupling

1. `RenderExtractorRegistry`,
2. feature plugins produkują `RenderFramePacket`,
3. WGPU renderer konsumuje packet zamiast service-per-domain coupling.

### Stream I - Diagnostics and Capabilities

1. `CapabilityRegistry` zamiast ręcznych list plugin/capability,
2. diagnostics/reporting czyta stan z registry,
3. usunięcie legacy `platformer_2d` po zamknięciu compatibility window.

## Definition of Done

1. `app` nie zawiera centralnych domenowych `match` dla scene/script commands.
2. nowy feature plugin można dodać bez edycji `crates/apps/app/src/lib.rs`.
3. renderer nie importuje bezpośrednio feature scene services.
4. `platformer_2d` nie jest już core capability.
5. sidescroller działa jako boilerplate/sample, nie jako engine special case.
6. testy bootstrap/integration/scripting przechodzą po każdym etapie.

2. nowy editor tooling,
3. duże zmiany wizualne/UI niezwiązane z architekturą runtime.
4. rename pakietu/crate/plugin identity `amigo-2d-platformer` jako osobny deprecation pass po Task 008.

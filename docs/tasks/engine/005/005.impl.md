# TASK 005 - Trait-First / Plugin-First / Generic-First Refactor

## Status

Completed

Progress note: this task is closed as a compatibility-preserving refactor slice. It does not mean the whole long-term architecture vision is finished; it means the scoped checkpoint described here is done and validated. Follow-up architectural work continues in later tasks, starting with `platformer -> motion`.

## Type

Architecture / Refactor / Engine Core

## Priority

High

## Goal

Refactor Amigo so the engine becomes:

1. trait-first
2. plugin-first
3. generic-first
4. game-agnostic

The engine must stop encoding gameplay genres like `platformer`, `sidescroller`, `rpg`, or `shooter` as core capabilities. Core engine responsibilities should stay neutral and composable:

1. scenes
2. entities
3. input
4. motion
5. physics
6. collisions
7. rendering
8. UI
9. audio
10. assets
11. scripting
12. hot reload
13. scheduling
14. diagnostics

Genre-specific behavior must move into:

1. boilerplates
2. sample mods
3. Rhai libraries
4. templates

## Architecture Test

The main success criteria for the whole refactor are:

```text
Adding a new feature does not require editing crates/apps/app/src/lib.rs.
Adding a new boilerplate does not require editing engine core.
Removing playground-sidescroller does not break the engine.
```

## Why This Task Exists

The current architecture works for feature incubation, but it is drifting toward a central orchestrator model:

1. `crates/apps/app/src/lib.rs` knows too many domains
2. `SceneCommand` is a central enum that knows many component types
3. the engine still exposes `platformer` as a first-class capability
4. ticking, asset synchronization, scripting bindings, and diagnostics are still too centralized

This task exists to reverse that direction before the current playground architecture hardens into engine policy.

## Refactor Rule

```text
Amigo does not implement game genres.
Amigo implements small neutral engine capabilities.
Genres and gameplay styles are assembled from those capabilities outside engine core.
```

## Non-Goals

This task does not aim to:

1. build a better sidescroller as a product feature
2. redesign the whole runtime in one PR
3. rewrite the engine into a different programming model
4. break all current playgrounds in the name of purity

The target is staged refactoring, not a flag day rewrite.

## Refactor Strategy

Proceed in small compatibility-preserving slices.

### Phase 1 - Mechanical App Split

Split [crates/apps/app/src/lib.rs](/D:/Git/amigo/crates/apps/app/src/lib.rs) without changing behavior.

Target layout:

```text
crates/apps/app/src/
  lib.rs
  bootstrap.rs
  summary.rs
  launch_selection.rs

  orchestration/
    mod.rs
    stabilize.rs
    placeholder_bridge.rs
    script_bridge.rs

    audio_bridge.rs
    hot_reload_bridge.rs

  scene_runtime/

    loading.rs
    hydration.rs
    transition.rs
    clear.rs

  assets/
    mod.rs

    physics_2d.rs
```

Target outcome:

1. `lib.rs` becomes a thin entrypoint
2. no behavior changes
3. test suite stays green

### Phase 2 - Generic Runtime Context

Build on `RuntimeContext` and use it consistently instead of large service parameter lists.

Direction:

1. `RuntimeContext`
2. `SceneCommandContext`
3. `ScriptCommandContext`
4. `SystemContext`
5. `AssetContext`
6. `RenderExtractContext`

Goal:

1. reduce massive parameter lists
2. centralize `required<T>()` / `optional<T>()`

### Phase 3 - Plugin Bundles

Add `PluginBundle` support to runtime construction.

Target:

```rust
let runtime = RuntimeBuilder::default()
    .with_bundle(CoreEngineBundle)?
    .with_bundle(PlatformWinitBundle)?
    .with_bundle(RenderWgpuBundle)?
    .with_bundle(TwoDBundle)?
    .with_bundle(ThreeDBundle)?
    .with_bundle(AudioBundle)?
    .with_bundle(UiBundle)?
    .with_bundle(ModdingBundle::new(options))?
    .with_bundle(RhaiScriptingBundle)?
    .build();
```

Goal:

1. stop growing a long `.with_plugin(...)` list in `app`
2. make engine assembly clearer and reusable

### Phase 4 - Neutralize `platformer` into `motion_2d`

Rename and reshape:

1. `crates/2d/platformer` -> `crates/2d/motion`
2. `PlatformerPlugin` -> `Motion2dPlugin`
3. `PlatformerSceneService` -> `Motion2dSceneService`
4. `PlatformerController2d` -> `MotionController2d` or `KinematicMotor2d`
5. `PlatformerControllerParams` -> `MotionProfile2d`
6. `PlatformerMotor2d` -> `MotionIntent2d`
7. `PlatformerFacing` -> `Facing2d`
8. `PlatformerAnimationState` -> `MotionAnimationState`
9. `platformer_2d` -> `motion_2d`
10. `world.platformer` -> `world.motion`

Keep:

1. physics in `amigo-2d-physics`
2. collisions in `amigo-2d-physics`
3. genre behavior outside core engine

1. `app` no longer centralizes all domain orchestration
2. new scene features do not require editing `crates/apps/app/src/lib.rs`

# TASK 009 - Zero-Legacy Motion Domain Cleanup

## Status

Planned.

## Type

Architecture Cleanup / Naming Finalization / Public API Tightening

## Priority

High

## Why This Task Exists

Task 008 zakończył ciężką część architektoniczną:

1. runtime przeszedł na aktywne `motion_2d`,
2. scene/runtime/scripting są w praktyce motion-first,
3. `playground-sidescroller` działa już jak sample mod używający engine capabilities.

Jednocześnie w repo nadal istnieją pozostałości starej tożsamości `platformer`:

- nazwa crate/package/plugin,
- aliasy `Platformer*`,
- alias parsera YAML,
- nazwy modułów i plików,
- reporting labels i testy kompatybilności.

Jeżeli obowiązuje zasada `zero legacy`, to te pozostałości nie są już “miłym do posprzątania detalem”, tylko ostatnim blokiem semantycznym oddzielającym engine od pełnej neutralności domenowej.

## Core Rule

`platformer` może istnieć wyłącznie jako:

1. gameplay sample,
2. boilerplate,
3. mod,
4. Rhai/gameplay library,
5. content vocabulary wewnątrz moda.

`platformer` nie może istnieć już jako:

1. nazwa capability engine,
2. nazwa pluginu engine,
3. nazwa crate domenowego engine,
4. publiczny typ domenowy engine,
5. aktywna ścieżka parsera YAML,
6. aktywna ścieżka API Rhai.

## Baseline Used For This Task

Task 009 jest oparty na:

1. aktualnym stanie repo,
2. audycie z `concat-output.txt`,
3. audycie trzech agentów obejmującym:
   - domenę ruchu,
   - scene/YAML/runtime path,
   - scripting/diagnostics/render/mod/docs.

To oznacza, że poniższa lista jest celowo konkretna i gotowa do wdrażania plik-po-pliku.

## Desired End State

Po zakończeniu Task 009:

1. engine posiada domenę `motion_2d`, a nie `platformer_2d`,
2. crate/package/plugin identity brzmi `amigo-2d-motion`,
3. publiczne typy engine są wyłącznie `Motion*`,
4. YAML przyjmuje wyłącznie `MotionController2D`,
5. Rhai ma wyłącznie `world.motion`,
6. diagnostics i summary nie emitują żadnej nazwy `amigo-2d-platformer`,
7. `playground-sidescroller` pozostaje sample modem zbudowanym z `motion_2d + physics_2d + camera_2d + tilemap_2d + ui + audio`.

## Explicit Non-Goals

Task 009 nie ma:

1. przebudowywać gameplayu sidescrollera,
2. zmieniać architektury motion/physics od strony zachowania,
3. ruszać asset pipeline poza mechaniką nazw, jeśli nie zdecydujemy się na content cleanup,
4. przepisywać historycznych tasków tylko po to, żeby nie zawierały słowa `platformer`,
5. robić kolejnego compatibility passu.

To jest task **breaking cleanup**, nie migration shim.

## What Belongs To Engine vs Boilerplate

### Engine

Engine powinien oferować tylko neutralne capability:

- `motion_2d`
- `physics_2d`
- `camera_follow_2d`
- `tilemap_2d`
- `sprite_2d`
- `text_2d`
- `vector_2d`
- `ui`
- `audio`
- `scene`
- `assets`
- `scripting`
- `render`

### Boilerplate / Gameplay Layer

Gameplay platformera/sidescrollera powinien żyć poza core engine:

- `mods/playground-sidescroller`
- przyszłe `boilerplates/sidescroller`
- Rhai libraries z gameplayowym klejem

### What The Sample Mod Is Allowed To Do

`playground-sidescroller` może nadal mówić:

- player
- jump
- coin
- finish
- HUD
- score
- platformer

bo to jest gameplay semantics moda, nie engine semantics.

## Workstream A - Rename The Motion Domain Crate

### Goal

Usunąć starą tożsamość `platformer` z manifestów i import paths.

### Files

- `Cargo.toml`
- `crates/2d/platformer/Cargo.toml`
- `crates/apps/app/Cargo.toml`
- `crates/scripting/rhai/Cargo.toml`
- `Cargo.lock`

### Actions

1. Rename workspace member:
   `crates/2d/platformer` -> `crates/2d/motion`
2. Rename package:
   `amigo-2d-platformer` -> `amigo-2d-motion`
3. Rename workspace dependency key:
   `amigo-2d-platformer` -> `amigo-2d-motion`
4. Update all downstream Cargo manifests.
5. Regenerate lockfile.

### Risks

1. Breakage in imports across `app`, `rhai`, tests.
2. Workspace dependency resolution fallout.

### Validation

- `cargo check`
- `cargo test -p amigo-2d-motion`
- `cargo test -p amigo-app`

## Workstream B - Remove Public Platformer Compatibility Surface

### Goal

Wyciąć z domeny ruchu wszystko, co nadal publicznie sugeruje istnienie `platformer`.

### Main File

- `crates/2d/motion/src/lib.rs`

### Remove

1. `LEGACY_PLATFORMER_PLUGIN_NAME`
2. `LEGACY_PLATFORMER_CAPABILITY`
3. `PlatformerDomainInfo`
4. All public aliases:
   - `PlatformerFacing`
   - `PlatformerAnimationState`
   - `PlatformerControllerParams`
   - `PlatformerControllerState`
   - `PlatformerMotor2d`
   - `PlatformerController2d`
   - `PlatformerController2dCommand`
   - `PlatformerDriveResult`
   - `PlatformerSceneService`
   - `PlatformerPlugin`
5. Legacy helper constructors/accessors:
   - `legacy_platformer_plugin`
   - `legacy_platformer_domain_info`
   - any `legacy_*` helper tied to the old name
6. Legacy-text error strings such as:
   `expect("platformer scene service mutex should not be poisoned")`

### Rewrite

1. Plugin name to canonical:
   `amigo-2d-motion`
2. Runtime reporting label to canonical motion identity only
3. Helper signatures that still mention `Platformer*` to `Motion*`

### Tests To Remove Or Rewrite

1. Legacy compatibility alias tests
2. Legacy plugin label mapping tests
3. Any test that proves old identity still works

### Rule

After this workstream, there must be **no public `Platformer*` symbol** in the motion domain.

## Workstream C - Remove Scene/YAML Legacy Parsing

### Goal

Engine YAML schema ma być wyłącznie motion-first.

### Files

- `crates/engine/scene/src/document.rs`
- `crates/engine/scene/src/lib.rs`

### Actions

1. Remove:
   `alias = "PlatformerController2D"`
2. Accept only:
   `MotionController2D`
3. Remove:
   `pub type PlatformerController2dSceneCommand = MotionController2dSceneCommand;`

### Tests

Delete or invert:

1. tests that parse `PlatformerController2D`
2. tests that verify old alias serializes back to motion

Recommended replacement:

1. parser accepts `MotionController2D`
2. parser rejects `PlatformerController2D`

## Workstream D - Rename Internal App Modules

### Goal

Usunąć mylące wewnętrzne nazwy `platformer2d`, skoro semantyka jest już motion-first.

### Files

- `crates/apps/app/src/scene_runtime/handlers/platformer2d.rs`
- `crates/apps/app/src/scene_runtime/handlers/mod.rs`
- `crates/apps/app/src/systems/motion_2d.rs`

### Actions

1. Rename file:
   `platformer2d.rs` -> `motion2d.rs`
2. Rename type:
   `ScenePlatformer2dCommandHandler` -> `SceneMotion2dCommandHandler`
3. Update handler registration imports.
4. Delete compatibility wrapper:
   `tick_platformer_world(...)`

### Rule

No active engine source file/module/type should still be named `platformer*`.

## Workstream E - Rename Rhai Motion Binding Module

### Goal

Rhai internals mają być spójne z publicznym API `world.motion`.

### Files

- `crates/scripting/rhai/src/bindings/platformer.rs`
- `crates/scripting/rhai/src/bindings/mod.rs`
- `crates/scripting/rhai/src/bindings/world_root.rs`
- `crates/scripting/rhai/src/lib.rs`

### Actions

1. Rename file:
   `bindings/platformer.rs` -> `bindings/motion.rs`
2. Rename module wiring:
   `mod platformer;` -> `mod motion;`
3. Re-export:
   `pub use motion::{MotionApi, MotionStateView};`
4. Update imports in `world_root.rs` and tests.
5. Update crate import path after crate rename.

### Rule

`world.motion` is already the public API. This workstream aligns internal naming with that fact.

## Workstream F - Fix Diagnostics / Summary / Reporting

### Goal

Public CLI/runtime output must stop leaking `amigo-2d-platformer`.

### Files

- `crates/apps/app/src/diagnostics.rs`
- `crates/apps/app/src/summary.rs`
- `crates/apps/app/src/lib.rs`
- `crates/2d/motion/src/lib.rs`

### Actions

1. Remove reporting based on:
   `motion_2d via amigo-2d-platformer`
2. Report canonical plugin/domain identity directly.
3. Rewrite test expectations that still rely on old label constants.

### Rule

If a user runs diagnostics after this task, they should never see `amigo-2d-platformer`.

## Workstream G - Test Cleanup

### Goal

Replace compatibility assertions with canonical-only assertions.

### Areas

- `crates/2d/motion/src/lib.rs`
- `crates/engine/scene/src/document.rs`
- `crates/scripting/rhai/src/lib.rs`
- `crates/apps/app/src/lib.rs`

### Actions

1. Delete tests that preserve old aliases.
2. Add tests that prove only canonical motion paths remain.
3. Ensure bootstraps and integration tests use `MotionController2D` and `world.motion`.

## Workstream H - Optional Content / Repo-Wide Wording Cleanup

### Goal

Only if desired: remove `platformer` wording from mod-local files and tools.

### Examples

- `mods/playground-sidescroller/tilesets/platformer.yml`
- `mods/playground-sidescroller/tilesets/platformer-rules.yml`
- `tools/generate-platformer-kit-assets.ps1`
- mod README wording

### Important

This is **not required** to make the engine zero-legacy.
It is a separate content/terminology decision.

2. Update imports/manifests

### Batch 2 - Remove Compatibility Layer

1. Remove `Platformer*` aliases from the motion crate

### Batch 3 - Scene/YAML Breaking Cleanup

## Definition of Done

Task 009 is complete when:

1. the engine crate is `amigo-2d-motion`,
2. runtime plugin identity is `amigo-2d-motion`,
3. there is no `platformer_2d` capability anywhere in active engine code,
4. there are no public `Platformer*` exports in active engine code,
5. YAML accepts only `MotionController2D`,
6. Rhai exposes only `world.motion`,
7. diagnostics/summary report only canonical motion identity,
8. active engine module/file/type names no longer use `platformer*`,
9. `playground-sidescroller` still boots and works as a sample mod built on engine features,
10. all targeted tests pass.

## Guidance For The Next Agent

The most important principle for the next implementation pass is this:

1. do not preserve aliases,
2. do not leave parser compatibility,
3. do not introduce deprecation shims,
4. do not treat `platformer` as a domain concept inside engine,
5. keep `platformer/sidescroller` semantics only where they belong: mod/sample/boilerplate layer.

If a rename feels “too breaking”, that is usually a sign the code still carries legacy coupling and the task is targeting exactly that coupling.

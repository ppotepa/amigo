# TASK 002 - Screen-Space UI as a runtime plugin

## Status

Planned

## Goal

Add the first native UI system to Amigo as a screen-space overlay rendered through WGPU, defined from scene YAML, and integrated with the current engine architecture:

```text
scene YAML -> scene hydration -> SceneCommand queue -> UI runtime services -> WGPU overlay rendering -> Rhai updates -> hot reload
```

The result of this task must be a working UI overlay in `playground-2d`, in a new scene:

`mods/playground-2d/scenes/screen-space-preview/scene.yml`

The `screen-space-preview` scene should demonstrate the current UI capabilities:

1. HUD panel
2. text
3. progress bar
4. button
5. simple Rhai-driven state updates

## Architectural Decision

UI is a separate engine domain. It is not part of:

1. `2d/text`
2. `2d/sprite`
3. `render-wgpu`
4. `app`

We add a new runtime plugin:

`amigo-ui`

This plugin registers UI runtime services and exposes the capability:

`screen_space_ui`

For this task, the only supported target is:

`screen-space`

Out of scope for this task:

1. WebView
2. HTML
3. world-space UI
4. billboard UI
5. surface UI
6. full CSS/flexbox
7. advanced focus/input routing

The system should stay deliberately simple, but structured so that future UI targets can be added cleanly.

## Current Engine Context

This task must fit the existing Amigo runtime pattern:

1. domains such as `sprite`, `text`, `mesh`, and `material` already register their own `SceneService` and `DomainInfo` through `RuntimePlugin`
2. the application composes runtime through `RuntimeBuilder::with_plugin(...)`
3. scene hydration already maps YAML components into `SceneCommand`
4. the runtime already supports scene-local `scene.rhai`, optional persistent `mod.rhai`, hot reload, and Rhai `world.*` APIs

UI must enter the architecture through the same runtime plugin path.

## Runtime Plugin

Add:

```rust
pub struct UiPlugin;

impl RuntimePlugin for UiPlugin {
    fn name(&self) -> &'static str {
        "amigo-ui"
    }

    fn register(&self, registry: &mut ServiceRegistry) -> AmigoResult<()> {
        registry.register(UiSceneService::default())?;
        registry.register(UiStateService::default())?;
        registry.register(UiInputService::default())?;
        registry.register(UiLayoutService::default())?;
        registry.register(UiDomainInfo {
            crate_name: "amigo-ui",
            capability: "screen_space_ui",
        })
    }
}
```

Then add:

```rust
.with_plugin(UiPlugin)?
```

to the app bootstrap next to the other domain plugins.

## Layout MVP

Supported node types:

1. `panel`
2. `column`
3. `row`
4. `stack`
5. `text`
6. `button`
7. `progress-bar`
8. `spacer`

Rules:

1. `screen-space` uses viewport pixels
2. root may define `left`, `top`, `width`, `height`
3. `column` stacks children vertically
4. `row` stacks children horizontally
5. `stack` overlays children
6. `padding` works on containers
7. `gap` works between children
8. `width` and `height` are optional
9. no percentage sizes in MVP
10. no full CSS layout model
11. no scrolling
12. no advanced clipping beyond a simple parent rect bound if implemented

Add:

```rust
pub struct UiRect {
    pub x: f32,
    pub y: f32,
    pub width: f32,
    pub height: f32,
}

pub struct UiLayoutNode {
    pub path: String,
    pub rect: UiRect,
    pub node: UiNode,
    pub children: Vec<UiLayoutNode>,
}
```

## WGPU Rendering

`WgpuSceneRenderer::render_scene(...)` must receive `UiSceneService`.

The renderer still draws world 2D and 3D first. UI is then added as an overlay.

For MVP, UI may render as:

1. colored quads
2. basic text using the existing placeholder text path
3. a simple progress bar

Conversion path:

```text
UiDocument
  -> UiLayoutTree
  -> UiDrawPrimitive
  -> ColorVertex
```

Primitive model:

```rust
pub enum UiDrawPrimitive {
    Quad {
        rect: UiRect,
        color: ColorRgba,
    },
    Text {
        rect: UiRect,
        content: String,
        color: ColorRgba,
        font_size: f32,
        font: Option<AssetKey>,
    },
    ProgressBar {
        rect: UiRect,
        value: f32,
        background: ColorRgba,
        foreground: ColorRgba,
    },
}
```

Render order:

1. clear background
2. 3D content
3. 2D content
4. text content
5. UI overlay

For MVP, `border_radius` may be stored in the model but ignored visually if necessary. The important part is that the format already contains the field.

## Rhai API

Add minimal API:

```rhai
world.ui.set_text("playground-2d-ui-preview.subtitle", "Updated from Rhai");
world.ui.set_value("playground-2d-ui-preview.hp-bar", 0.5);
world.ui.show("playground-2d-ui-preview.root");
world.ui.hide("playground-2d-ui-preview.root");
```

Add a scene script:

`mods/playground-2d/scenes/screen-space-preview/scene.rhai`

Example:

```rhai
let square = world.entities.named("playground-2d-ui-preview-square");
let hp = 0.75;

fn on_enter() {
    world.dev.event("playground-2d.ui-preview.entered", world.scene.current_id());
    world.ui.set_text("playground-2d-ui-preview.subtitle", "Rhai connected");
}

fn update(dt) {
    square.rotate_2d(dt);

    if world.input.down("ArrowLeft") {
        hp = hp - dt * 0.25;
    }

    if world.input.down("ArrowRight") {
        hp = hp + dt * 0.25;
    }

    if hp < 0.0 {
        hp = 0.0;
    }

    if hp > 1.0 {
        hp = 1.0;
    }

    world.ui.set_value("playground-2d-ui-preview.hp-bar", hp);
}

fn on_event(topic, payload) {
    if topic == "playground-2d.ui-preview.button-clicked" {
        world.dev.log("UI preview button clicked");
        world.ui.set_text("playground-2d-ui-preview.subtitle", "Button clicked");
    }
}
```

If Rhai global mutation makes this awkward, HP state may be held in `UiStateService` or simplified, but the acceptance criteria still require that Rhai can call `world.ui.set_text` and `world.ui.set_value`.

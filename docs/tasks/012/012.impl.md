# TASK 012: Split `playground-2d-asteroids` into Menu, Options, Highscores, and Game Scenes

## Goal

Split the current single-scene Asteroids redesign into a cleaner multi-scene arcade flow:

- `main-menu`
- `options`
- `highscores`
- `game`

The current `mods/playground-2d-asteroids/scenes/vector-preview` scene now contains both real gameplay and screen routing. That was useful for Task 011, but it is the wrong long-term shape. The next cleanup is to move non-gameplay UI flow out of the gameplay scene.

The goal is not to add Asteroids-specific engine systems. The goal is a reusable pattern for scene routing, session state, UI documents, and game-scene isolation.

## Current State

Task 011 introduced:

- YAML main menu, highscores, game-over, HUD, and center status UI.
- Procedural asteroid waves.
- Neutral `asteroids` pool with `asteroid-01..asteroid-16`.
- Bullet pool and projectile emitter.
- `LOW` mode.
- Session highscores stored in scene state.
- Runtime fixes for UI root visibility.

Remaining structural problem:

- `scene.rhai` still owns menu routing, options-like LOW mode, highscores, game-over, and gameplay in one script.
- `scene.yml` still contains UI documents for every screen in the same scene.
- `LOW` mode and highscores are scene-local state, so a naive scene split would lose them on scene transition.

## Target Scene Layout

```text
mods/playground-2d-asteroids/scenes/
  main-menu/
    scene.yml
    scene.rhai

  options/
    scene.yml
    scene.rhai

  highscores/
    scene.yml
    scene.rhai

  game/
    scene.yml
    scene.rhai
```

`vector-preview` can either:

- be removed/renamed after the split, if no external tests depend on it; or
- temporarily become a compatibility alias scene that selects `main-menu`.

Prefer explicit new scene IDs over overloading `vector-preview`.

## Responsibility Split

### `main-menu`

Owns:

- title screen UI
- menu navigation
- selected item highlight
- routing to `game`, `options`, `highscores`
- `QUIT` fallback if app quit API is unavailable

Must not own:

- gameplay entities
- asteroid pools
- score/lives/wave rules
- procedural wave generation

### `options`

Owns:

- LOW intensity mode toggle
- future toggles such as audio, debug grid, input help, accessibility

Initial options:

```text
LOW: OFF/ON
BACK
```

Must persist `LOW` across scene changes.

### `highscores`

Owns:

- session highscore display
- back navigation to `main-menu`

Initial version can keep session-only highscores. Do not implement persistent storage in this task unless it already exists as a generic service.

### `game`

Owns only gameplay:

- new run initialization
- score/lives/wave
- ship spawn/respawn
- shield timer
- procedural wave generation
- bullet/asteroid collision events
- ship/asteroid collision events
- game over handling

Must not own:

- main menu navigation
- options screen
- highscores screen UI

Game over can be implemented in either of two acceptable ways:

1. Keep a small in-game game-over overlay inside `game`, then route back to `main-menu`.
2. Add a separate `game-over` scene later.

For this task, prefer option 1 to avoid over-splitting.

## Required Engine/API Design

### 1. Add Session State

The key blocker for scene split is state lifetime.

Existing `SceneStateService` is scene-scoped and should reset on scene reload/transition. That is correct for gameplay values like `score`, `lives`, and `wave`.

Add a generic session-level state service:

```rust
pub enum SessionValue {
    Int(i64),
    Float(f64),
    Bool(bool),
    String(String),
}

pub struct SessionStateService {
    values: Mutex<BTreeMap<String, SessionValue>>,
}
```

Rhai API:

```rhai
world.session.set_bool("asteroids.low_mode", true);
world.session.get_bool("asteroids.low_mode");

world.session.set_int("asteroids.highscore.1", 10000);
world.session.get_int("asteroids.highscore.1");
```

Rules:

- Generic name: `SessionStateService`, not `AsteroidsState`.
- Persists across scene changes.
- Clears only when runtime/session resets, not when a scene changes.
- Supports the same primitive types as `SceneStateService`.
- Namespacing is caller-owned, e.g. `asteroids.low_mode`.

Suggested paths:

```text
crates/gameplay/state/src/lib.rs
crates/scripting/rhai/src/bindings/session.rs
crates/scripting/rhai/src/bindings/mod.rs
```

If `crates/gameplay/state` already owns scene state/timers, extend it there. Do not create a duplicate state crate.

Acceptance:

- `LOW` survives `main-menu -> options -> main-menu -> game`.
- Highscores survive `game -> main-menu -> highscores`.
- Scene reload still clears scene-local score/lives/wave.

### 2. Scene Routing

Use existing scene selection API if available:

```rhai
world.scene.select("game");
world.scene.select("options");
world.scene.select("highscores");
world.scene.select("main-menu");
```

If current API names scenes by startup scene IDs only, align scene IDs with directory names:

```yaml
scene:
  id: main-menu
```

Acceptance:

- `START` from menu routes to `game`.
- `OPTIONS` routes to `options`.
- `HIGHSCORES` routes to `highscores`.
- `BACK`/`Escape` routes back to `main-menu`.

### 3. UI Visibility

Important implementation note from Task 011 bugfix:

UI overlay visibility is controlled by `UiStateService` paths such as:

```text
playground-2d-asteroids-main-menu.root
```

Do not rely only on:

```rhai
world.entities.hide("playground-2d-asteroids-main-menu");
```

For in-scene UI documents, use:

```rhai
world.ui.show("playground-2d-asteroids-hud.root");
world.ui.hide("playground-2d-asteroids-hud.root");
```

Scene hydration should continue to initialize root UI visibility from entity `visible`.

Acceptance:

- Hidden UI documents do not render.
- Starting gameplay does not leave menu overlay on screen.
- Tests assert `UiStateService::is_visible("...root")`, not only `SceneService::is_visible(entity)`.

## Scene Details

## `main-menu/scene.yml`

Contains:

- black/retro background
- optional subtle vector title decoration
- `UiDocument` for menu
- audio references for `menu-move`, `menu-select`

Menu items:

```text
START
OPTIONS
HIGHSCORES
QUIT
```

Use `text_bind` keys:

```yaml
text_bind: menu_start_text
text_bind: menu_options_text
text_bind: menu_highscores_text
text_bind: menu_quit_text
```

## `main-menu/scene.rhai`

Responsibilities:

```rhai
fn on_enter() {
    world.state.set_int("menu_index", 0);
    sync_menu(world);
}

fn update(dt) {
    update_menu_input(world);
}
```

Routing:

```rhai
if item == "START" {
    world.scene.select("game");
}

if item == "OPTIONS" {
    world.scene.select("options");
}

if item == "HIGHSCORES" {
    world.scene.select("highscores");
}
```

If `world.app.quit()` does not exist:

```rhai
world.state.set_string("status_text", "QUIT UNAVAILABLE");
```

Do not add `world.app.quit()` unless it is generic and cleanly supported by host runtime.

## `options/scene.yml`

Contains:

- `UiDocument` with title `OPTIONS`
- `LOW: ON/OFF`
- `BACK`

Use `text_bind`:

```yaml
text_bind: option_low_text
text_bind: option_back_text
```

## `options/scene.rhai`

Responsibilities:

- read/write `world.session.bool("asteroids.low_mode")`
- update selected item text
- back to `main-menu`

Example:

```rhai
fn toggle_low(world_api) {
    let current = world_api.session.get_bool("asteroids.low_mode");
    world_api.session.set_bool("asteroids.low_mode", !current);
    sync_options(world_api);
}
```

Acceptance:

- LOW mode text updates immediately.
- LOW mode persists after returning to menu and starting game.

## `highscores/scene.yml`

Contains:

- `UiDocument` with title `HIGHSCORES`
- highscore list text
- back prompt

Use `text_bind`:

```yaml
text_bind: highscores_text
```

## `highscores/scene.rhai`

Responsibilities:

- render highscores from `world.session`
- return to `main-menu`

Initial keys:

```text
asteroids.highscore.1
asteroids.highscore.2
asteroids.highscore.3
asteroids.highscore.4
asteroids.highscore.5
```

If keys are missing, seed default values in `main-menu` or `highscores`.

## `game/scene.yml`

Contains:

- gameplay background
- ship
- shield
- bullet pool
- asteroid pool
- HUD
- center status
- game-over overlay, if kept in-game
- collision events
- audio assets/cues

Move from current `vector-preview/scene.yml`:

- `arena-void`
- stars/frame if used
- ship
- shield
- bullet entities
- asteroid entities
- `EntityPool` components
- `ProjectileEmitter2D`
- `Velocity2D`
- `Bounds2D`
- `Lifetime`
- collision events
- HUD and status UI

Remove from game scene:

- main menu UI
- options UI
- highscores UI

## `game/scene.rhai`

Responsibilities:

- start a new run in `on_enter`
- read `LOW` from session
- manage `score`, `lives`, `wave`
- generate waves
- handle collision events
- update HUD
- route to `main-menu` after game over

Important: `game` scene should not have a `game_mode == "menu"` branch.

Acceptable game-local modes:

```text
playing
wave_clear
game_over
```

Optional:

```text
respawn
```

Example:

```rhai
fn on_enter() {
    world.state.reset_scene();
    world.timers.reset_scene();
    start_new_game(world);
}

fn low_mode(world_api) {
    return world_api.session.get_bool("asteroids.low_mode");
}

fn update(dt) {
    if world.input.any_pressed(["Escape"]) {
        world.scene.select("main-menu");
        return;
    }

    if world.state.get_string("game_mode") == "playing" {
        update_playing(world);
        return;
    }

    if world.state.get_string("game_mode") == "wave_clear" {
        update_wave_clear(world);
        return;
    }

    if world.state.get_string("game_mode") == "game_over" {
        update_game_over(world);
        return;
    }
}
```

On final game over:

```rhai
fn enter_game_over(world_api) {
    maybe_insert_session_highscore(world_api, world_api.state.get_int("score"));
    world_api.state.set_string("game_mode", "game_over");
    world_api.ui.show("playground-2d-asteroids-game-over.root");
}
```

Back to menu:

```rhai
if world_api.input.any_pressed(["Space", "Enter", "Escape"]) {
    world_api.scene.select("main-menu");
}
```

## Tests

## Unit Tests

Add/extend tests for session state:

- `session_state_persists_across_scene_state_reset`
- `session_state_get_missing_values_returns_defaults`
- `session_state_supports_int_float_bool_string`
- Rhai binding test for `world.session.set/get/add` where applicable

## Runtime Tests

Update Asteroids runtime tests:

1. Bootstrap starts at `main-menu`.
2. Main menu root UI is visible.
3. HUD/gameplay UI is not visible in main menu.
4. `Space` on `START` selects `game`.
5. Game scene shows ship/HUD and hides menu.
6. `Escape` from game returns to `main-menu`.
7. `OPTIONS` scene toggles LOW.
8. LOW persists into `game`.
9. Game over inserts score into session highscores.
10. `HIGHSCORES` scene renders session highscores.

Important test assertion:

```rust
assert!(!ui_state.is_visible("playground-2d-asteroids-main-menu.root"));
```

Do not only assert:

```rust
assert!(!scene.is_visible("playground-2d-asteroids-main-menu"));
```

The latter does not prove UI overlay is hidden.

## Implementation Order

1. Add generic `SessionStateService`.
2. Add Rhai `world.session` binding.
3. Add tests for session state.
4. Create `main-menu` scene from current menu UI.
5. Create `options` scene and move LOW toggle there.
6. Create `highscores` scene and move highscore UI there.
7. Create `game` scene from current gameplay entities/HUD/collision events.
8. Strip menu/options/highscores branches from gameplay Rhai.
9. Wire scene routing.
10. Update startup/tests from `vector-preview` to `main-menu`, or add alias scene.
11. Run targeted tests.
12. Cleanup old `vector-preview` content if compatibility allows.

## Definition Of Done

- [ ] Asteroids startup scene is `main-menu` or `vector-preview` aliases to `main-menu`.
- [ ] `main-menu` has no gameplay entities/pools.
- [ ] `options` owns LOW mode UI.
- [ ] `highscores` owns highscore display UI.
- [ ] `game` owns gameplay entities, pools, collision events, HUD, and game-over overlay.
- [ ] `game/scene.rhai` has no main menu navigation branch.
- [ ] `game/scene.rhai` has no highscores screen branch.
- [ ] LOW mode persists across scene transitions.
- [ ] Highscores persist across scene transitions for the current runtime session.
- [ ] Starting game hides menu overlay through `UiStateService`.
- [ ] Tests assert UI root visibility.
- [ ] No new Asteroids-specific engine systems.
- [ ] Any new engine API is generic and reusable.

## Non-Goals

- Persistent highscores on disk.
- Full options menu framework.
- Save profiles.
- Controller rebinding UI.
- `world.app.quit()` unless already supported cleanly.
- Separate `game-over` scene, unless it falls out naturally.

# TASK 004 - TileRuleSet2D / AutoTile2D

## Status

Completed

## Type

Feature / Engine Domain / Content Pipeline

## Priority

High

## Goal

Add a reusable autotiling system for `TileMap2D` so scenes can define **logical terrain** with a simple grid, while the engine resolves the correct **visual tile variants** automatically from neighbor relationships.

The target is not a one-off helper for the current sidescroller. It is a general 2D world-building feature that future mods can reuse for:

1. platform edges
2. inner platform spans
3. vertical columns
4. top and bottom caps
5. later corners and richer terrain transitions

At the end of the MVP, a grid like:

```yaml
grid:
  - "........"
  - "..####.."
```

should no longer render as four identical blocks. It should resolve to:

1. `left_cap`
2. `middle`
3. `middle`
4. `right_cap`

with collision still driven by the logical grid rather than the chosen visual sprite variant.

## Why This Task Exists

Right now `TileMap2D` can parse a symbolic grid and build a playable world, but it still treats terrain too literally:

1. `#` means one tile id
2. repeated terrain cells look repetitive
3. visual edges of platforms are not expressed automatically
4. the engine cannot distinguish between a platform cap and an interior fill tile

That is enough for a bootstrap playground, but it is not enough for maintainable content authoring.

What we want instead:

1. scenes define the **shape** of terrain
2. rulesets define how that shape maps to tile variants
3. the engine resolves tile variants automatically
4. collision remains tied to logical terrain, not decorative tile selection

This task is about introducing that separation cleanly.

## Core Architectural Rule

Separate:

1. **logical terrain**
2. **visual variant resolution**

That means:

1. the tilemap grid says what kind of world cell exists
2. the ruleset says how to draw it
3. collision and gameplay depend on the logical cell
4. visual tile choice depends on neighboring cells

Do not make collision depend on whether a tile resolved to `left_cap`, `middle`, `top_cap`, or `corner`.

## Core Concepts

### 1. Logical TileMap2D

`TileMap2D` remains the owner of:

1. tile grid symbols
2. terrain meaning
3. marker extraction
4. collision source generation

### 2. TileRuleSet2D

A new asset kind describing how logical terrain symbols resolve into visual tile ids.

### 3. ResolvedTileMap2D

A runtime-produced structure containing final tile ids chosen by the ruleset for rendering.

### 4. Tile Resolver

A deterministic runtime utility that:

1. reads logical grid
2. checks neighbors
3. chooses the best variant
4. emits resolved visual tiles

## Scope of MVP

MVP should support:

1. one terrain family like `ground`
2. 4-direction neighbor analysis:
   1. top
   2. right
   3. bottom
   4. left
3. horizontal platform variants:
   1. `single`
   2. `left_cap`
   3. `middle`
   4. `right_cap`
4. schema that already leaves room for:
   1. `top_cap`
   2. `bottom_cap`
   3. `vertical_middle`
   4. inner corners
   5. outer corners

Important:

The engine should compute full 4-direction neighborhood data from the beginning, even if the first playable use only consumes horizontal variants.

That is the intended forward-compatible escape hatch.

## Non-Goals for MVP

Do not implement all of these in the first step:

1. 8-neighbor blob autotiling
2. Wang tiles
3. slope logic
4. biome blending
5. random tile variation
6. weighted decoration placement
7. LDtk/Tiled import of autotile metadata
8. editor tooling
9. parallax terrain materials

## New Asset Kind

Add:

`tile-ruleset-2d`

This should be parsed by the asset pipeline like other structured assets.

Suggested metadata file:

`mods/<mod>/tilesets/<name>-rules.yml`

Example:

```yaml
kind: tile-ruleset-2d
label: Platformer Ground Rules
format: amigo-rules-v1
tile_size:
  x: 16
  y: 16

terrains:
  ground:
    symbol: "#"
    collision: solid

    variants:
      single: 1
      left_cap: 2
      middle: 3
      right_cap: 4

      top_cap: 5
      bottom_cap: 6
      vertical_middle: 7

      inner_corner_top_left: 8
      inner_corner_top_right: 9
      inner_corner_bottom_left: 10
      inner_corner_bottom_right: 11

      outer_corner_top_left: 12
      outer_corner_top_right: 13
      outer_corner_bottom_left: 14
      outer_corner_bottom_right: 15
```

In MVP, many of those fields may remain optional or unused, but the format must already allow them.

## Runtime Model

Suggested internal types:

```rust
pub struct TileRuleSet2D {
    pub tile_size: UVec2,
    pub terrains: Vec<TerrainRule2D>,
}

pub struct TerrainRule2D {
    pub id: String,
    pub symbol: char,
    pub collision: TileCollisionKind,
    pub variants: TileVariantMap2D,
}

pub struct TileVariantMap2D {
    pub single: Option<u32>,
    pub left_cap: Option<u32>,
    pub middle: Option<u32>,
    pub right_cap: Option<u32>,
    pub top_cap: Option<u32>,
    pub bottom_cap: Option<u32>,
    pub vertical_middle: Option<u32>,
    pub inner_corner_top_left: Option<u32>,
    pub inner_corner_top_right: Option<u32>,
    pub inner_corner_bottom_left: Option<u32>,
    pub inner_corner_bottom_right: Option<u32>,
    pub outer_corner_top_left: Option<u32>,
    pub outer_corner_top_right: Option<u32>,
    pub outer_corner_bottom_left: Option<u32>,
    pub outer_corner_bottom_right: Option<u32>,
}

pub struct TileNeighbors2D {
    pub top: bool,
    pub right: bool,
    pub bottom: bool,
    pub left: bool,
}

pub struct ResolvedTile2D {
    pub terrain_id: String,
    pub source_symbol: char,
    pub tile_id: u32,
    pub neighbors: TileNeighbors2D,
}

pub struct ResolvedTileMap2D {
    pub width: usize,
    pub height: usize,
    pub tiles: Vec<Option<ResolvedTile2D>>,
}
```

## Scene Schema Direction

`TileMap2D` should gain a `ruleset` reference.

Example:

```yaml
- type: TileMap2D
  tileset: playground-sidescroller/tilesets/platformer
  ruleset: playground-sidescroller/tilesets/platformer-rules
  tile_size: { x: 16.0, y: 16.0 }
  grid:
    - "................................................................"
    - ".................###............................................"
    - ".....P............................C.........................F..."
    - "############################..###############################..."
```

Behavior:

1. `grid` still defines logical cells
2. `ruleset` selects visual variants
3. collision still comes from logical terrain semantics
4. render uses resolved variants

## Resolution Logic

For each logical terrain cell:

1. find its terrain definition by symbol
2. inspect top/right/bottom/left neighbors of the same terrain kind
3. build a neighborhood mask
4. resolve a visual variant

### MVP horizontal rules

Suggested rule priority:

1. no left, no right -> `single`
2. no left, yes right -> `left_cap`
3. yes left, yes right -> `middle`
4. yes left, no right -> `right_cap`

This is enough for simple platform strips.

### Forward-compatible vertical and corner rules

Even if not used yet in current content, the resolver should be structured so these can be added without redesign:

1. `top_cap`
2. `bottom_cap`
3. `vertical_middle`
4. `inner_corner_*`
5. `outer_corner_*`

That means:

1. do not hardcode only left/right checks in the main model
2. compute all 4 cardinal neighbors up front
3. keep variant selection in one dedicated resolver function

## Collision Rule

Collision is derived from logical terrain cells, not from resolved visual tile ids.

Example:

1. `#` remains solid
2. regardless of whether it resolves to:
   1. `single`
   2. `left_cap`
   3. `middle`
   4. `right_cap`

This avoids tight coupling between art and physics.

The asset pipeline should:

Hydration should include that reference in `TileMap2dSceneCommand`.

### 4. `crates/apps/app`

At hydration/runtime time:

1. register the ruleset asset

Tilemap rendering should:

1. use resolved tile ids
2. compute UV from final tile id
3. stay compatible with current textured tilemap path

## MVP Flow

```text
TileMap2D scene component
-> logical grid
-> ruleset asset reference
-> prepared tile-ruleset-2d
-> resolver computes final visual tile ids
-> collision generated from logical terrain
-> renderer draws resolved variants from tileset
```

## Testing Plan

### Tile ruleset parsing

1. parse `tile-ruleset-2d`
2. parse terrain symbol mapping
3. parse optional variant fields

### Resolver tests

1. `"#"` resolves to `single`
2. `"##"` resolves to `left_cap`, `right_cap`
3. `"###"` resolves to `left_cap`, `middle`, `right_cap`
4. vertical stack test reserved by schema, even if visual rules are partial
5. missing optional variants fail gracefully or fall back predictably

### Collision tests

1. logical `#` still produces solid collision regardless of visual variant
2. collision output remains stable when ruleset changes only visual tile ids

### Hydration / bootstrap tests

1. `TileMap2D` with `ruleset` parses from scene YAML
2. ruleset asset is registered/prepared
3. bootstrap summary contains no failed assets
4. resolved tilemap is produced for runtime use

### Hot reload tests

1. modifying ruleset metadata triggers reprepare
2. scene reload updates rendered variant selection
3. collision remains stable unless logical grid changes

## Final Intent

This task should establish a clean world-building pattern for Amigo:

1. scenes describe world shape simply
2. rulesets describe visual interpretation
3. engine resolves tile variants
